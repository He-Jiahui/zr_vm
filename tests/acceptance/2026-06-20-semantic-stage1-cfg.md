# Semantic Stage 1 CFG Reachability

## Scope
- Start Stage 1 from `docs/plans/lsp/01-semantic-inference-core.md` with the smallest CFG-backed semantic fact slice.
- Affected layers: parser CFG scaffold, semantic reachability fact emission, parser test target registration, and semantic documentation.

## Baseline
- `SZrSemanticReachabilityFact` already had statement-level causes such as `AFTER_RETURN`, but there was no parser CFG producer.
- The first Stage 1 slice was intentionally narrow: straight-line statement lists plus terminator statements.
- The next Stage 1 CFG slices add `if` statement branch graph support for boolean literal and folded unary/logical boolean conditions, short-circuit decisive operands, integer literal comparisons, first switch case body graph support, boolean/integer/string/char/float switch case/default pruning, folded boolean selector/case matching for switch, mixed-kind known constant switch case pruning, constant-matched switch no-default fallthrough pruning, `while(false)` and folded-false loop-body pruning, first traditional `for(false)` body pruning, first `foreach` body graph support, first try/catch/finally body graph support, throw/call-gated catch body filtering, catch-all ordering including untyped `catch (e)`, known literal throw kind set, explicit cast throw expression, sequential local declaration typed catch matching, same-block simple assignment typed catch matching, simple same-name if/else branch-assignment typed catch matching, one-sided branch assignment plus incoming type typed catch matching, simple multi-variable branch assignment typed catch matching, simple unknown-condition while/traditional-for body, traditional-for init, and traditional-for step assignment typed catch matching, straight-line terminator-aware result-binding pruning after `return`/`throw`/`break`/`continue`, and folded selected-branch terminator sibling pruning for typed catch matching, basic `break`/`continue` target edges for supported while/for/foreach loops, `return`/`throw`/`break`/`continue` abrupt paths from try/catch into finally, separated normal/function-exit/break/continue finally paths, and removal of fake zero-iteration exits for constant-true `while`/traditional `for` plus omitted-condition `for(;;)`. Broader expression folding, more complete typed catch type/pattern matching beyond known literal/cast/local-declaration/straight-line-assignment/simple full-branch, partial-branch, multi-variable branch assignment, simple unknown-loop init/body/step assignment kinds, or proven terminator pruning, finer function-effect/nonthrow metadata, full switch/union exhaustiveness, and broader loop fixed-point/nontermination precision remain pending. Switch-local break was audited and is not a current CFG target because the compiler only resolves break/continue through loop labels.
- This continuation additionally covers simple conditional `continue` transfer bindings for post-loop typed-catch matching: an assignment before `continue` is preserved as a possible post-loop value after a later condition exit, and traditional `for` continue paths apply step bindings before merge.
- This continuation additionally covers folded `if` selection while collecting loop-control transfer bindings: an unreachable constant branch with its own `break` or `continue` no longer contributes stale assignment kinds to post-loop typed-catch matching.
- This continuation additionally covers folded `if` selection while proving switch case/default body fallthrough for post-switch typed-catch matching: a selected folded branch that terminates does not let assignments from that branch or an unselected branch pollute the later `throw value`.
- This continuation additionally covers direct lambda IIFE throw-effect profiling: a primary lambda expression immediately called by a single function-call member now contributes only its call arguments/generic arguments and lambda body to catch reachability and known throw-kind collection. General named/member/import/construct calls remain conservative.
- This continuation additionally covers mixed-kind known constant equality/inequality in the shared condition folder: `if (1 == "1")` is now deterministically false, while relational mixed-kind comparisons, implicit conversions, numeric ranges, symbolic values, and arithmetic folding remain outside this slice.
- This continuation additionally covers known integer addition in the shared condition folder: `if ((1 + 1) == 2)` is now deterministically true after checked integer `+` folding. Overflowed arithmetic, division, modulo, implicit conversions, numeric ranges, and symbolic values remain outside this slice.
- This continuation additionally covers known integer subtraction in the shared condition folder: `if ((3 - 1) == 2)` is now deterministically true after checked integer `-` folding. Division, modulo, implicit conversions, numeric ranges, and symbolic values remain outside this slice.
- This continuation additionally covers known integer multiplication in the shared condition folder: `if ((2 * 3) == 6)` is now deterministically true after checked integer `*` folding. Division, modulo, implicit conversions, numeric ranges, and symbolic values remain outside this slice.
- This continuation additionally covers known integer division and modulo in the shared condition folder: `if ((9 / 3) == 3)` and `if ((10 % 4) == 2)` are now deterministically true after checked integer `/` and `%` folding. Division/modulo by zero and the signed `INT64_MIN / -1` or `INT64_MIN % -1` overflow/undefined-behavior boundaries remain unknown; implicit conversions, numeric ranges, symbolic values, and floating arithmetic remain outside this slice.
- This continuation additionally covers finite float addition in the shared condition folder: `if ((1.5 + 2.25) == 3.75)` is now deterministically true after finite float `+` folding. Non-finite operands/results, implicit conversions, numeric ranges, and symbolic values remain outside the current accepted surface.

Non-finite operands/results, implicit conversions, numeric ranges, and symbolic values remain outside the current accepted surface.

Negative bitwise operands, `^`, shifts, implicit conversions, numeric ranges, mixed-type arithmetic, and symbolic values remain outside the current accepted surface.

Negative bitwise operands, shifts, implicit conversions, numeric ranges, mixed-type arithmetic, and symbolic values remain outside the current accepted surface.
- This continuation additionally covers known nonnegative integer left shift in the shared condition folder: `if ((3 << 2) == 12)` is now deterministically true after binary `<<` folding. Negative operands, negative or too-large shift counts, overflowed results, implicit conversions, numeric ranges, mixed-type arithmetic, and symbolic values remain outside the current accepted surface.

Negative operands, negative or too-large shift counts, implicit conversions, numeric ranges, mixed-type arithmetic, and symbolic values remain outside the current accepted surface.
- This continuation additionally covers same-kind folded constant relational comparisons in the shared condition folder: `if ((1 + 2) > 2)` and `if ((1.5 + 2.25) > 3.0)` are now deterministically true after the folded constants are compared. Mixed-kind relational comparisons, string/bool relational operators, implicit conversions, numeric ranges, symbolic values, and more complex combination expressions remain outside this slice.
- The semantic query diagnostics bridge now consumes unreachable reachability facts and exposes them as structured `unreachable_code` warnings. LSP/compiler frontend publishing of those query diagnostics remains pending.

- `tests/parser/test_cfg_reachability.c`

  - The emitted reachability fact keeps cause `ZR_SEMANTIC_REACHABILITY_AFTER_RETURN` and points `causeNode` at the return statement.
  - A `return` terminator block connects to exit while the following statement remains entry-unreachable.
  - Two expression statements in sequence produce no unreachable fact.

## Acceptance Decision
- Accepted for Stage 1 CFG straight-line, bool-literal and unary/logical/short-circuit folded boolean `if`/loop conditions, integer/string/char/float literal comparison conditions, first switch-case body graph, boolean/integer/string/char/float switch case/default pruning, folded boolean switch selector/case matching, mixed-kind known constant switch case pruning, known matching switch no-default fallthrough pruning, `while(false)`, first traditional `for(false)` body-pruning, first `foreach` body graph, first try/catch/finally body graph, throw/call-gated catch body filtering, catch-all ordering including untyped `catch (e)`, known literal throw kind set, explicit cast throw expression, sequential local declaration type binding, same-block simple assignment type binding, simple same-name if/else branch-assignment type binding, simple one-sided branch-assignment plus incoming type binding, simple multi-variable branch-assignment result binding, simple unknown-condition `while` body assignment plus incoming type binding, simple unknown-condition traditional `for` body assignment plus incoming type binding, simple traditional `for` init assignment before condition handling, simple unknown-condition traditional `for` step assignment plus incoming type binding, straight-line terminator-aware result-binding pruning after `return`/`throw`/`break`/`continue`, and folded selected-branch terminator sibling pruning for typed catch matching, try/catch `return`/`throw`/`break`/`continue` paths into finally, single and mixed normal/break/continue paths through finally back to their correct targets, basic while/for/foreach `break`/`continue` target-edge slices, and removal of synthetic zero-iteration exits for known-true `while`/traditional `for` and omitted-condition `for(;;)`.
- Accepted additionally for simple conditional `continue` transfer bindings feeding post-loop typed-catch matching, with traditional `for` step binding applied to continue transfers before merge.
- Accepted additionally for folded `if` selection during loop-control transfer binding collection, so only the selected constant branch contributes `break`/`continue` transfer assignments to post-loop typed-catch matching.
- Accepted additionally for folded `if` selection during switch case/default fallthrough proof, so only the selected constant branch decides whether a case/default body can contribute assignments to post-switch typed-catch matching.
- Accepted additionally for simple symbolic `if` fallthrough result binding: assignments in a branch that terminates with `return`/`throw`/`break`/`continue` no longer contribute to post-branch typed-catch matching, while the non-terminating sibling branch remains precise.
- Accepted additionally for nested conditional `break` transfer prefix bindings in typed-catch loop flow: assignments already active before a nested `break` branch now remain part of the possible post-loop `throw value` kind set.
- Accepted additionally for direct lambda IIFE throw-effect profiling: a directly visible nonthrowing lambda IIFE can make a following catch body unreachable, and a directly visible throwing lambda IIFE can contribute its lambda-body builtin throw kind set to typed-catch ordering.
- Accepted additionally for mixed-kind known constant equality/inequality condition folding: known constants of different literal kinds are deterministic nonmatches for `==` and deterministic matches for `!=`, feeding `if`/loop condition reachability through the shared condition folder.
- Accepted additionally for known integer addition condition folding: checked integer-literal `+` expressions can fold to integer constants before equality/relational comparison, feeding `if`/loop reachability through the shared condition folder.
- Accepted additionally for known integer subtraction condition folding: checked integer-literal `-` expressions can fold to integer constants before equality/relational comparison, feeding `if`/loop reachability through the shared condition folder.
- Accepted additionally for known integer multiplication condition folding: checked integer-literal `*` expressions can fold to integer constants before equality/relational comparison, feeding `if`/loop reachability through the shared condition folder.
- Accepted additionally for known integer division/modulo condition folding: checked integer-literal `/` and `%` expressions can fold to integer constants before equality/relational comparison when they avoid zero divisors and the signed `INT64_MIN / -1` / `INT64_MIN % -1` boundary, feeding `if`/loop reachability through the shared condition folder.
- Accepted additionally for finite float addition condition folding: finite float-literal `+` expressions can fold to float constants before equality/relational comparison when both operands and the result are finite, feeding `if`/loop reachability through the shared condition folder.
- Accepted additionally for finite float subtraction condition folding: finite float-literal `-` expressions can fold to float constants before equality/relational comparison when both operands and the result are finite, feeding `if`/loop reachability through the shared condition folder.
- Accepted additionally for finite float multiplication condition folding: finite float-literal `*` expressions can fold to float constants before equality/relational comparison when both operands and the result are finite, feeding `if`/loop reachability through the shared condition folder.
- Accepted additionally for finite float division condition folding: finite float-literal `/` expressions can fold to float constants before equality/relational comparison when both operands are finite, the divisor is nonzero, and the result is finite, feeding `if`/loop reachability through the shared condition folder.
- Accepted additionally for integer and finite float unary-minus condition folding: known integer-literal unary `-` expressions can fold with int64 overflow checks, and finite float-literal unary `-` expressions can fold when operands/results remain finite, feeding `if`/loop reachability through the shared condition folder.
- Accepted additionally for integer and finite float unary-plus condition folding: known integer-literal and finite float-literal unary `+` expressions preserve their operand constants before equality/relational comparison, feeding `if`/loop reachability through the shared condition folder.
- Accepted additionally for integer bitwise-not condition folding: known integer-literal unary `~` expressions fold through the int64 identity `~x == -1 - x`, feeding `if`/loop reachability through the shared condition folder without depending on C signed bit representation.
- Accepted additionally for nonnegative integer bitwise-AND condition folding: known nonnegative integer-literal binary `&` expressions fold before equality/relational comparison, feeding `if`/loop reachability through the shared condition folder while negative operands remain unknown.
- Accepted additionally for nonnegative integer bitwise-OR condition folding: known nonnegative integer-literal binary `|` expressions fold before equality/relational comparison, feeding `if`/loop reachability through the shared condition folder while negative operands remain unknown.
- Accepted additionally for nonnegative integer bitwise-XOR condition folding: known nonnegative integer-literal binary `^` expressions fold before equality/relational comparison, feeding `if`/loop reachability through the shared condition folder while negative operands remain unknown.
- Accepted additionally for nonnegative integer left-shift condition folding: known nonnegative integer-literal binary `<<` expressions fold before equality/relational comparison when the shift count is valid and the result stays in int64 range, feeding `if`/loop reachability through the shared condition folder while negative operands/counts and overflowed results remain unknown.
- Accepted additionally for nonnegative integer right-shift condition folding: known nonnegative integer-literal binary `>>` expressions fold before equality/relational comparison when the shift count is valid, feeding `if`/loop reachability through the shared condition folder while negative operands/counts remain unknown.
- Accepted additionally for same-kind folded constant relational condition comparison: folded integer, finite float, and char constants can feed `<`, `>`, `<=`, and `>=` decisions after existing arithmetic/unary folding, while mixed-kind relational comparisons and string/bool relational operators remain unknown.
- Stage 1 remains open for more complete typed catch type/pattern matching beyond known literal/cast/local-declaration/straight-line-assignment/simple full-branch, partial-branch, multi-variable branch, simple unknown-while assignment, simple unknown-for init/body/step assignment, and proven terminator-pruned kind sets; finer function-effect/nonthrow metadata; union switch/default matching; broader constant-condition folding beyond literal/unary/logical short-circuit/comparison, mixed-kind equality expressions, checked integer addition/subtraction/multiplication/division/modulo/unary plus/minus/bitwise not/nonnegative bitwise AND/OR/XOR/left-shift/right-shift, finite float addition/subtraction/multiplication/division/unary plus/minus, and same-kind folded integer/float/char relational comparisons; broader loop fixed-point/nontermination precision including symbolic conditional break/continue-sensitive typed catch exits and break-before-exit assignment flow; concrete dataflow analyses; semantic-query local re-analysis; and LSP/compiler frontend serialization/publishing of semantic query diagnostics.
