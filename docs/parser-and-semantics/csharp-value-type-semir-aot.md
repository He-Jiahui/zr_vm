# C#-Style Value-Type SemIR

This contract records the first typed value-place layer used to move `struct` toward C# value-type semantics and IL2CPP-style lowering.

## Scope

`struct` locals that have `ZR_FUNCTION_FRAME_SLOT_KIND_INLINE_STRUCT` frame layout are treated as inline value payloads for SemIR metadata. The interpreter still executes the existing bytecode path, but SemIR now preserves the value-type intent so AOT lowering does not have to infer it from untyped stack mutations.

The initial opcode family is:

- `VALUE_ADDR`: address/place for a frame value slot.
- `FIELD_ADDR`: address/place for a field inside an inline struct receiver.
- `LOAD_VALUE`: typed load from a value place into a destination slot.
- `STORE_VALUE`: typed store into a value place.
- `INIT_VALUE`: reserved for layout-aware zero/default construction.
- `COPY_VALUE`: layout-aware by-value copy between inline struct slots.
- `CALL_TYPED`: typed call metadata for known call sites whose return destination has proven inline struct layout.
- `RETURN_TYPED`: typed return metadata for single-result inline struct return sources.

## Current Lowering

`compiler_semir.c` maps known inline struct bytecode patterns into metadata rows:

- `GET_MEMBER_SLOT` on an inline struct receiver emits `FIELD_ADDR` followed by `LOAD_VALUE`.
- `SET_MEMBER_SLOT` on an inline struct receiver emits `FIELD_ADDR` followed by `STORE_VALUE`.
- `SET_STACK` between inline struct slots emits `COPY_VALUE`; if the source was staged through the immediately preceding `GET_STACK`, the original inline struct source slot is recorded.
- `FUNCTION_CALL` / `KNOWN_VM_CALL` with a result slot that has proven inline struct layout emits `CALL_TYPED`. The destination slot is the return payload slot, `operand0` is the callee slot, and `operand1` is the bytecode argument count.
- Single-result `FUNCTION_RETURN` from a proven inline struct source emits `RETURN_TYPED`. The destination slot and `operand0` both identify the source return payload slot.

`FIELD_ADDR.operand0` is the receiver slot. `FIELD_ADDR.operand1`, `LOAD_VALUE.operand1`, and `STORE_VALUE.operand1` carry the member/cache index used by the source bytecode. `COPY_VALUE.operand0` is the source inline struct slot. Call-result temporaries can recover `typeTableIndex` through matching inline `typeLayoutId` frame-slot layout when there is no direct typed-local binding for the temporary slot.

## AOT Boundary

The AOT ExecIR name table recognizes the new opcodes. These value-place opcodes currently add no runtime contract bits because they are intended to lower into direct frame/field/copy expressions once AotExecIR consumes layout metadata.

The archived AotExecIR frame layout now mirrors inline frame byte metadata:

- `SZrAotExecIrFrameLayout.frameByteSize`
- `SZrAotExecIrFrameLayout.frameByteAlign`
- `SZrAotExecIrFrameLayout.slotLayoutCount`
- `SZrAotExecIrFrameLayout.slotLayouts`

Each `SZrAotExecIrFrameSlotLayout` copies stack slot, byte offset, byte size, byte align, type layout id, slot kind, and parameter marker from `SZrFunctionFrameSlotLayout`. This gives future generated C lowering a direct route from `FIELD_ADDR` / `LOAD_VALUE` / `STORE_VALUE` / `COPY_VALUE` to frame byte spans without reverse-engineering layout from flat stack counts.

The archived `aot_c` writer now has a first value-place lowering surface in `backend_aot_c_value_semir.*` plus the focused field module `backend_aot_c_value_semir_fields.*`. `backend_aot_write_c_value_semir_for_function()` receives `SZrState`, walks each function's AotExecIR SemIR rows, finds frame-slot byte layout with `backend_aot_c_find_frame_slot_layout()`, and dispatches field address/load/store rows to the field module. The field module resolves struct field metadata with `ZrCore_Function_ResolvePrototypeFrameFieldLayout()` and emits value-place lowering annotations for:

- `FIELD_ADDR` -> `zr_aot_value_field_addr`
- `LOAD_VALUE` -> `zr_aot_value_load`
- `STORE_VALUE` -> `zr_aot_value_store`
- `COPY_VALUE` -> `zr_aot_value_copy`
- `CALL_TYPED` -> `zr_aot_value_call_typed`
- `RETURN_TYPED` -> `zr_aot_value_return_typed`

For equal-sized copy spans it records the current generated-frame ABI shape: `memmove((TZrByte *)frame.slotBase + dstOffset, (const TZrByte *)frame.slotBase + srcOffset, size)`. Field address, field load, and field store annotations now record `base byte offset + field byte offset` expressions after resolving the member/cache index to `SZrFunctionFrameFieldLayout`. Typed call/return annotations record the involved frame layout and type id, and the first POD executable path now consumes the same layout facts.

`backend_aot_c_value_semir.c` is now the value-SemIR orchestrator and copy/call/return lowering surface. Field-specific helpers, including primitive field transfer, embedded `SZrTypeValue` transfer, layout-aware nested inline struct transfer, and the explicit unsupported resolved-field boundary, live in `backend_aot_c_value_semir_fields.c`. This keeps the value-place field responsibility below the large-file threshold and gives future field policies a narrow module boundary instead of growing the orchestrator.

The first executable M5 steps are now inline `COPY_VALUE` lowering and primitive POD field load/store lowering. `backend_aot_try_write_c_value_semir_for_exec_instruction()` checks the current exec instruction for a matching value SemIR row:

- `COPY_VALUE` on matching inline struct `SET_STACK` / `GET_STACK` sites now emits a generated type-layout gate and skips the old generic stack-copy helper. The generated C resolves `ZrCore_Function_ResolvePrototypeFrameTypeLayout(frame.function, typeLayoutId, state)`, uses `ZrCore_TypeLayout_CanRawCopy()` to keep POD struct copies on `memmove`, and uses `ZrCore_TypeLayout_CopyInline()` for non-POD field-aware copies such as structs with embedded `SZrTypeValue` fields.
- `LOAD_VALUE` on matching inline struct `GET_MEMBER_SLOT` sites resolves the field layout, reads the primitive field bytes from `(const TZrByte *)frame.slotBase + baseOffset + fieldOffset`, writes the scalar result into the byte-backed destination `SZrTypeValue`, and mirrors that result into the dense `frame.slotBase + destinationSlot` view. Generated code may still have direct typed arithmetic, branch, or return paths that read the dense view while the byte-frame ABI migration is incomplete, so field loads must keep both views coherent.
- `STORE_VALUE` on matching inline struct `SET_MEMBER_SLOT` sites resolves the field layout, prefers a populated dense `frame.slotBase + sourceSlot` source over the byte-backed source view, converts the scalar `SZrTypeValue` to the field primitive type, and copies the primitive bytes into `(TZrByte *)frame.slotBase + baseOffset + fieldOffset`. This prevents stale byte-backed scratch values from overriding newer direct arithmetic results written to the dense slot.
- `LOAD_VALUE` / `STORE_VALUE` on matching inline struct fields that are stored as embedded `SZrTypeValue` cells now emit `zr_aot_value_exec_field_value_slot_load` / `zr_aot_value_exec_field_value_slot_store` and use `ZrCore_Value_Copy` directly between the frame byte field and the scalar destination/source slot. Value-slot loads also mirror the copied field into the dense destination, and value-slot stores prefer the dense source when it is materialized. This is the first AOT managed/reference field access slice; copy/drop ownership for whole non-POD structs still belongs to later layout-aware copy/drop lowering.
- `LOAD_VALUE` / `STORE_VALUE` on matching nested inline struct fields now emit `zr_aot_value_exec_field_inline_struct_load` / `zr_aot_value_exec_field_inline_struct_store` when the field `typeLayoutId` and byte size match the inline destination/source slot. The generated C resolves the field `SZrTypeLayout` with `ZrCore_Function_ResolvePrototypeFrameTypeLayout(frame.function, typeLayoutId, state)`, checks the emitted field byte size, keeps POD field layouts on overlap-safe `memmove`, and routes non-POD field layouts through `ZrCore_TypeLayout_CopyInline(state, ...)`. This mirrors whole-struct `COPY_VALUE` semantics for nested value fields instead of unconditionally raw-copying embedded ownership/value payloads.
- When `LOAD_VALUE` / `STORE_VALUE` resolves an inline struct field but the field is still outside the executable subset, for example a mismatched nested inline struct transfer, the generated C emits `zr_aot_value_unsupported_field_load` / `zr_aot_value_unsupported_field_store`, calls `ZrCore_Debug_RunError(state, "unsupported AOT value SemIR field ...")`, and exits through `ZR_AOT_C_FAIL()`. Unresolved or dynamic field access still falls through to the declared runtime member contract; proven value-type fields do not silently fall back to `GetMemberSlot` / `SetMemberSlot`.
- `CALL_TYPED` on matching static/direct POD struct return sites resolves the caller destination type layout with `ZrCore_Function_ResolvePrototypeFrameTypeLayout`, checks the generated-frame destination byte size, stages the callable and arguments at `ZrCore_Function_GetCallInfoFrameStorageTop(state, frame.callInfo)`, resolves callable metadata from the callee slot with `ZrCore_Closure_GetMetadataFunctionFromValue`, prepares the resolved VM call with `ZrCore_Function_PreCallPreparedResolvedVmFunctionWithArgumentSource`, invokes the generated callee thunk, posts through `ZrCore_Function_PostCall`, and refreshes the active caller frame. Passing the caller `frame.slotBase` and source argument start lets the core runtime copy byte-backed VALUE parameters from the original generated frame while still using the staged call window for VM call setup.
- `RETURN_TYPED` on matching single inline struct return sites resolves the callee source type layout with `ZrCore_Function_ResolvePrototypeFrameTypeLayout`, checks the inline source byte size, publishes the source slot as `state->stackTop`, and lets core post-call route the payload through `ZrCore_Function_TryCopyInlineFrameReturnValue`.

Generated AOT C functions now also have a first value-frame cleanup boundary. The generated guard macro uses `ZR_AOT_C_RETURN(expr)` instead of emitting direct `return` statements from normal return, tail return, unsupported-dispatch, and failure paths. Each generated function records `zr_aot_frame_started`, a `zr_aot_return_value`, and `zr_aot_skip_drop_slot`, then exits through `zr_aot_function_exit`. `backend_aot_c_frame_cleanup.*` emits reverse frame-layout cleanup for inline struct slots: it resolves each slot's `SZrTypeLayout` from `frame.function`, skips the typed inline return source slot when that source must remain available for core post-call return movement, and calls `ZrCore_TypeLayout_DropInline` only when the resolved layout has a non-`NONE` drop kind. This does not complete hidden-return ownership or exception-aware partial initialization; it prevents the generated C path from bypassing the existing layout-driven drop contract once non-POD inline value slots are active.

Generated AOT C failure exits now report directly from the generated guard macro. `ZR_AOT_C_FAIL()` calls `ZrCore_Debug_RunError(state, "generated AOT function failed: functionIndex=%u instructionIndex=%u", ...)` with deterministic frame metadata and then exits through `ZR_AOT_C_RETURN(0)`. Generated functions declare `ZrAotGeneratedFrame frame = {0};`, so failures that happen before generated-frame begin setup still have a defined frame state. The checked C backend no longer emits `ZrLibrary_AotRuntime_FailGeneratedFunction(state, &frame)`; the remaining generated-C runtime boundary in `backend_aot_c*.c` is the private loaded-module context lookup.

Generated AOT C frame setup now owns simple setup diagnostics and observation policy initialization directly. `backend_aot_write_c_frame_setup()` still calls `ZrLibrary_AotRuntime_ResolveGeneratedModuleContext(state, functionIndex, &zr_aot_context)` because loaded AOT records, function tables, generated thunk tables, module publication state, and cached frame slot counts are private runtime-loader metadata, not instruction execution behavior. The public context type is named `ZrAotGeneratedModuleContext` to keep that boundary explicit. After that boundary, generated C validates the active call frame, reports `generated AOT function has no call frame` through `ZrCore_Debug_RunError`, initializes frame slot spans directly, reads override state from `state->hasAotObservationPolicyOverride`, `state->aotObservationMask`, and `state->aotPublishAllInstructions`, emits the default observation mask as an explicit OR of `ZR_AOT_GENERATED_STEP_FLAG_*` flags, and reads `ZR_DEBUG_HOOK_MASK_LINE` directly. The active runtime no longer exposes `ZrLibrary_AotRuntime_ReportGeneratedContextError()`, and generated C frame setup no longer emits `ZrLibrary_AotRuntime_GetGeneratedContext()`, `ZrLibrary_AotRuntime_GetObservationPolicy()`, `ZrLibrary_AotRuntime_DefaultObservationMask()`, or `ZrLibrary_AotRuntime_BeginGeneratedFunction()`. `ZrLibrary_AotRuntime_GetObservationPolicy()` and `ZrLibrary_AotRuntime_DefaultObservationMask()` remain available for tests and non-generated policy queries.

Generated AOT C root/export returns now publish module exports directly in generated code instead of calling `ZrLibrary_AotRuntime_PublishModuleExports(state, &frame)`. `ZrAotGeneratedFrame` carries the active `module`, `moduleExecuted` flag, function table, and generated thunk table from frame setup. `backend_aot_write_c_publish_exports()` emits `zr_aot_publish_exports_direct`, validates `frame.module`, `frame.moduleExecuted`, `frame.function`, and `frame.slotBase`, iterates `frame.function->exportedVariables`, reads each value from `frame.slotBase + export.stackSlot`, copies non-callable values directly, constructs callable export `SZrClosureNative` values directly from metadata and generated thunk pointers, binds captured values from frame/native/VM closure sources when available, calls `ZrCore_Module_AddPubExport` or `ZrCore_Module_AddProExport`, and finally sets `*frame.moduleExecuted = ZR_TRUE`. Generated modules now include `zr_vm_core/module.h` and `zr_vm_common/zr_ast_constants.h` for the direct publication loop. The checked C backend no longer emits `ZrLibrary_AotRuntime_PublishModuleExports(state, &frame)` or `ZrLibrary_AotRuntime_MaterializeModuleExportValue(...)`, and the stale public export-materialization wrapper was removed from the active AOT runtime API.

Generated AOT C no longer depends on a `.codex`-local verification copy of its module ABI. `zr_vm_common/include/zr_vm_common/zr_aot_abi.h` is now the public include-tree contract for generated modules and the runtime loader. It defines `ZR_VM_AOT_ABI_VERSION`, `EZrAotBackendKind`, `EZrAotInputKind`, `FZrAotEntryThunk`, `ZrAotCompiledModule`, `FZrVmGetAotCompiledModule`, and `ZR_VM_AOT_EXPORT`, matching the declarations emitted by `backend_aot_c_emitter.c`. `zr_vm_parser/include/zr_vm_parser/writer.h` now also owns `SZrAotWriterOptions` plus the AOT C/LLVM writer entrypoint declarations, so archived backend syntax checks and external writer consumers no longer need a local typedef for the writer options shape. This makes generated C syntax checks use the same public include surface that downstream C compilation must consume.

Primitive constant materialization now has a direct generated-C write path for the immediate scalar subset. `GET_CONSTANT` for null, bool, signed integer, unsigned integer, and float constants emits `zr_aot_value_exec_primitive_constant`, resolves the destination `SZrTypeValue *`, releases any existing owned/GC payload in that slot, and writes the new scalar with `ZrCore_Value_ResetAsNull` or `ZR_VALUE_FAST_SET(zr_aot_destination, ...)`. It no longer builds a temporary `zr_aot_constant` and copies it into the destination with `ZrCore_Value_Copy`. `SET_CONSTANT` in generated AOT C now emits `zr_aot_value_exec_set_constant` and writes `*zr_aot_constant = *zr_aot_source`, matching the interpreter's current raw constant-table assignment instead of calling `ZrLibrary_AotRuntime_SetConstant`. Resolved callable constants, zero-capture `CREATE_CLOSURE` opcodes, and zero-capture `GET_SUB_FUNCTION` child functions construct native closures directly from metadata and generated thunk pointers. `GET_SUB_FUNCTION` emits `zr_aot_value_exec_get_sub_function_native_closure`, validates the child metadata, allocates `SZrClosureNative`, assigns `zr_aot_closure->nativeFunction = zr_aot_fn_N`, and records callable provenance for the destination only on that direct path. Unresolved callable `GET_CONSTANT` materialization emits `zr_aot_value_unsupported_callable_constant_materialization`, validates the source constant and destination slot, checks callable metadata, reports `unsupported AOT callable constant materialization`, and fails through `ZR_AOT_C_FAIL()` instead of calling `ZrLibrary_AotRuntime_CopyConstant`. Captured or unresolved `CREATE_CLOSURE` now emits `zr_aot_value_unsupported_create_closure_materialization`, records `zr_aot_capture_count`, validates the source/destination/metadata boundary, reports `unsupported AOT CREATE_CLOSURE materialization`, clears callable provenance for that destination, and fails explicitly instead of calling `ZrLibrary_AotRuntime_CreateClosure`. Captured, unresolved, or missing `GET_SUB_FUNCTION` child closures emit `zr_aot_value_unsupported_get_sub_function_materialization`, record the child capture count when known, report `unsupported AOT GET_SUB_FUNCTION materialization`, clear callable provenance, and fail explicitly instead of calling `ZrLibrary_AotRuntime_GetSubFunction`. Plain non-immediate constants still follow the direct constant-table `ZrCore_Value_Copy` path; LLVM keeps its old helper routes until parity and real captured-closure materialization are handled.

The focused M2 scalar-local path extends immediate primitive constants with destination-local mirrors. `backend_aot_write_c_direct_primitive_constant()` receives `functionIr`, writes the existing frame-slot `SZrTypeValue`, then checks scalar-local coverage and mirrors bool/i64/u64/f64 immediates into `zr_aot_bN`, `zr_aot_sN`, `zr_aot_uN`, or `zr_aot_fN` when the constant payload type matches the declared local kind. The typed-scalar generated product now locks signed and float mirrors such as `zr_aot_s0 = (TZrInt64)21;`, `zr_aot_s6 = (TZrInt64)9;`, `zr_aot_f19 = (TZrFloat64)1.5;`, and `zr_aot_f20 = (TZrFloat64)2;`. Current `uint` literals in the focused source still materialize first as signed `INT64` constants and are then carried by later typed unsigned operations, so this slice deliberately does not claim direct `uN` literal mirroring for that frontend shape.

The focused M2 branch path now distinguishes proven slot/slot predicates from unproven slot/constant predicates. For the straight-line `if (product > threshold)` / `JUMP_IF_LESS_EQUAL_SIGNED` shape, both operands are written earlier in the same basic block, so generated C can use the maintained `zr_aot_s2` / `zr_aot_s4` locals directly after validating the frame slots. For the later `if (product == 42)` / `JUMP_IF_NOT_EQUAL_SIGNED_CONST` shape, the current block does not prove `product` was written before the branch, so generated C conservatively keeps the local scalar temporary `zr_aot_left_scalar` instead of assuming `zr_aot_s2` is authoritative. This keeps the current focused branch optimization correct while leaving cross-block local-state reconstruction, resume/deopt local restoration, and full C-local authoritative storage as later AOT work.

Closure captured-value access now has direct generated-C lowering for `GET_CLOSURE`, `SET_CLOSURE`, `GETUPVAL`, and `SETUPVAL`. Generated C emits `zr_aot_value_exec_get_closure_value` / `zr_aot_value_exec_set_closure_value`, resolves the current closure from `ZrCore_Stack_GetValue(frame.slotBase - 1)` after validating `frame.slotBase`, handles native closures through `ZrCore_ClosureNative_GetCaptureValue`, handles VM closures through `ZrCore_ClosureValue_GetValue`, copies values with `ZrCore_Value_Copy`, and preserves setter write barriers with `ZrCore_Value_Barrier`. The C backend no longer emits `ZrLibrary_AotRuntime_GetClosureValue` or `ZrLibrary_AotRuntime_SetClosureValue` for these opcodes. The generated-C smoke is compile-only for this hand-built opcode fixture; executing real captured closure AOT paths remains tied to captured-closure materialization and direct call work. LLVM parity remains future work.

Ownership opcode lowering now has direct generated-C core calls for `OWN_UNIQUE`, `OWN_BORROW`, `OWN_LOAN`, `OWN_SHARE`, `OWN_WEAK`, `OWN_DETACH`, `OWN_UPGRADE`, and `OWN_RELEASE`. Generated C emits `zr_aot_value_exec_ownership_core` for copy-like ownership operations, validates `state`, `frame.function`, `frame.slotBase`, and `frame.generatedFrameSlotCount`, resolves destination/source `SZrTypeValue` operands with `ZrCore_Stack_GetValue(frame.slotBase + slot)`, then calls `ZrCore_Ownership_UniqueValue`, `BorrowValue`, `LoanValue`, `ShareValue`, `WeakValue`, `DetachValue`, or `UpgradeValue` directly. Failed ownership transfer resets the destination with `ZrCore_Value_ResetAsNull`, matching the old wrapper behavior without preserving the instruction-helper call boundary. `OWN_RELEASE` emits `zr_aot_value_exec_ownership_release`, calls `ZrCore_Ownership_ReleaseValue(state, zr_aot_source)`, and resets the destination slot to null. The C backend no longer emits `ZrLibrary_AotRuntime_Own*` for these ownership opcodes; LLVM parity and source-level executable ownership fixtures remain future work.

Super-array integer opcodes now lower in generated AOT C through direct core fast semantic boundaries instead of instruction helper wrappers. `SUPER_ARRAY_GET_INT`, `SUPER_ARRAY_GET_INT_PLAIN_DEST`, `SUPER_ARRAY_SET_INT`, `SUPER_ARRAY_ADD_INT`, `SUPER_ARRAY_ADD_INT4`, `SUPER_ARRAY_ADD_INT4_CONST`, and `SUPER_ARRAY_FILL_INT4_CONST` are routed through `backend_aot_c_lowering_super_array.c`, resolve stack slots from `frame.slotBase`, validate signed integer operands and constants, then call the fast core APIs declared in `zr_vm_core/object.h`: `ZrCore_Object_SuperArrayTryGetIntFast`, `TrySetIntFast`, `AddIntAssumeFast`, `AddIntDiscardResultAssumeFast`, `AddInt4ConstAssumeFast`, and `FillInt4ConstAssumeFast`. The generated path deliberately avoids the generic `ZrCore_Object_SuperArrayGetInt` / `SetInt` / `AddInt` wrappers because those may fall back to dynamic member/index behavior. Non-fast or malformed shapes report `unsupported AOT super-array integer fast path` and fail through `ZR_AOT_C_FAIL()` instead of emitting `ZrLibrary_AotRuntime_SuperArray*`; LLVM parity and broader dynamic super-array lowering remain future work.

Iterator opcodes now lower in generated AOT C through direct core iterator boundaries instead of instruction helper wrappers. `ITER_INIT`, `DYN_ITER_INIT`, `ITER_MOVE_NEXT`, `DYN_ITER_MOVE_NEXT`, `ITER_CURRENT`, and `SUPER_DYN_ITER_MOVE_NEXT_JUMP_IF_FALSE` are routed through `backend_aot_c_lowering_iterators.c`, resolve iterator/source/destination stack slots from `frame.slotBase`, then call `ZrCore_Object_IterInit`, the cached array fast path `ZrCore_Object_TryIterMoveNextCachedArrayFast`, fallback `ZrCore_Object_IterMoveNext`, the cached current fast path `ZrCore_Object_TryIterCurrentCachedMemberFastStackResult`, or fallback `ZrCore_Object_IterCurrent` directly. The fused move-next false branch validates the result as bool and emits the generated `goto` directly, so it no longer routes through `ZrLibrary_AotRuntime_IsTruthy`. Unsupported iterator shapes report `unsupported AOT iterator core path` or `unsupported AOT iterator branch result` and fail through `ZR_AOT_C_FAIL()` instead of emitting `ZrLibrary_AotRuntime_Iter*`; LLVM parity and broader source-level iterator execution fixtures remain future work.

Scope lifecycle opcodes now lower in generated AOT C through direct frame, stack, and closure core operations. `MARK_TO_BE_CLOSED` emits `zr_aot_scope_mark_to_be_closed`, validates `frame.slotBase` and the generated-frame slot count, then calls `ZrCore_Closure_ToBeClosedValueClosureNew(state, frame.slotBase + slot)`. `CLOSE_SCOPE` emits `zr_aot_scope_close_scope`, preserves `state->stackTop` with `ZrCore_Stack_SavePointerAsOffset` and `ZrCore_Stack_LoadOffsetToPointer`, raises stack top to the active call frame top when needed, then closes pending scope registrations with `ZrCore_Closure_CloseStackValue` and `ZrCore_Closure_CloseRegisteredValues` directly in generated C. The C backend no longer emits `ZrLibrary_AotRuntime_MarkToBeClosed` or `ZrLibrary_AotRuntime_CloseScope` for these opcodes; LLVM parity remains helper-backed until that backend receives the same direct-core contract.

Dynamic/meta value access now has an explicit generated-C failure boundary for `META_GET`, `META_SET`, `SUPER_META_GET_CACHED`, `SUPER_META_SET_CACHED`, `SUPER_META_GET_STATIC_CACHED`, and `SUPER_META_SET_STATIC_CACHED`. Generated C emits `zr_aot_value_unsupported_meta_value_access`, materializes the primary stack operand, secondary stack operand, and member/cache index, reports `unsupported AOT meta value access` with `ZrCore_Debug_RunError`, and exits through `ZR_AOT_C_FAIL()` instead of calling `ZrLibrary_AotRuntime_Meta*` instruction helpers. This records object/meta member access as an unsupported dynamic boundary until real generated meta value access and meta dispatch semantics are specified. LLVM keeps its old helper routes until parity is handled.

Meta-call opcodes now also have an explicit generated-C unsupported boundary. `SUPER_META_CALL_NO_ARGS`, `SUPER_META_CALL_CACHED`, `SUPER_META_TAIL_CALL_NO_ARGS`, `SUPER_META_TAIL_CALL_CACHED`, `META_CALL`, and `META_TAIL_CALL` route through `backend_aot_write_c_unsupported_meta_call()`, which emits `zr_aot_unsupported_meta_call`, records destination, receiver, and argument count slots, validates the generated frame slots, reports `unsupported AOT meta call`, and exits through `ZR_AOT_C_FAIL()`. The C backend no longer emits `ZrLibrary_AotRuntime_PrepareMetaCall`, `ZrLibrary_AotRuntime_CallPreparedOrGeneric`, or `ZrAotGeneratedDirectCall zr_aot_direct_call` for these meta-call bytecodes. Real generated meta dispatch remains a future contract; this slice prevents it from being hidden behind the old prepared-call helper route.

Unsupported generated-instruction paths now use the same explicit generated-C failure model. Invalid branch targets, default opcode lowering paths, dispatch-loop fallthrough, and generated function fallthrough route through `backend_aot_write_c_unsupported_instruction()` / `_expr()`, emit a `zr_aot_unsupported_instruction` marker with function index, instruction index, and opcode metadata, report `unsupported AOT instruction`, and exit through `ZR_AOT_C_FAIL()`. The C backend no longer emits `ZrLibrary_AotRuntime_ReportUnsupportedInstruction` from `backend_aot_c*.c`; unsupported opcodes remain unsupported, but they are no longer hidden behind an instruction-helper call.

Generated AOT C instruction-entry bookkeeping is now emitted directly. `backend_aot_write_c_begin_instruction()` emits a `zr_aot_begin_instruction` block that refreshes `frame.callInfo`, `frame.slotBase`, `state->callInfoList`, and `state->stackTop` from the active call frame, assigns `frame.currentInstructionIndex`, and inlines the observation/line-debug branch that publishes `programCounter`, `lastObservedInstructionIndex`, `lastObservedLine`, and `ZrCore_Debug_Hook` line events when the generated step flags or dynamic line hook require it. The C backend no longer emits `ZrLibrary_AotRuntime_BeginInstruction` from `backend_aot_c*.c`; LLVM still uses the helper until LLVM parity is handled.

Pending control transfer opcodes now lower through generated C control-flow blocks instead of instruction helper wrappers. `SET_PENDING_RETURN`, `SET_PENDING_BREAK`, and `SET_PENDING_CONTINUE` route through `backend_aot_write_c_pending_control_transfer()`, emit `zr_aot_pending_return` / `zr_aot_pending_break` / `zr_aot_pending_continue` markers, validate the active generated frame, optionally resolve the pending return value from `frame.slotBase`, call `execution_set_pending_control`, then either resume an outer finally with `execution_resume_pending_via_outer_finally` or jump to the target instruction with `execution_jump_to_instruction_offset`. The block refreshes `frame.callInfo`, `frame.slotBase`, `state->callInfoList`, and `state->stackTop`, recomputes `zr_aot_next_instruction` from the active program counter, and dispatches inside the generated function. The C backend no longer emits `ZrLibrary_AotRuntime_SetPendingReturn`, `ZrLibrary_AotRuntime_SetPendingBreak`, or `ZrLibrary_AotRuntime_SetPendingContinue` from `backend_aot_c*.c`; LLVM parity and full generated control/exception execution fixtures remain separate work.

`TRY` and `END_TRY` now lower through direct generated C exception-handler stack operations instead of instruction helper wrappers. `TRY` emits `zr_aot_try_direct`, validates the active generated frame and handler index, resolves the active `SZrCallInfo`, calls `execution_push_exception_handler(state, zr_aot_call_info, handlerIndex)`, and refreshes `frame.callInfo`, `frame.slotBase`, `state->callInfoList`, and `state->stackTop`. `END_TRY` emits `zr_aot_end_try_direct`, finds the active handler with `execution_find_handler_state(state, zr_aot_call_info, handlerIndex)`, reads `frame.function->exceptionHandlerList[handlerIndex]`, either marks the handler state as `ZR_VM_EXCEPTION_HANDLER_PHASE_FINALLY` or pops it with `execution_pop_exception_handler`, and refreshes the generated frame. The C backend no longer emits `ZrLibrary_AotRuntime_Try` or `ZrLibrary_AotRuntime_EndTry` from `backend_aot_c*.c`.

`CATCH` now lowers through direct generated C exception materialization instead of the instruction helper wrapper. The generated `zr_aot_catch_direct` block validates the destination stack slot against `frame.generatedFrameSlotCount`, resolves it with `ZrCore_Stack_GetValue(frame.slotBase + slot)`, copies `state->currentException` into the destination with `ZrCore_Value_Copy` and clears it with `ZrCore_Exception_ClearCurrent` when present, writes null with `ZrCore_Value_ResetAsNull` when no current exception exists, and clears pending control with `execution_clear_pending_control`. Generated modules include `zr_vm_core/exception.h` for the exception clear API. The C backend no longer emits `ZrLibrary_AotRuntime_Catch` from `backend_aot_c*.c`.

`THROW` now lowers through direct generated C exception normalization and in-frame unwind dispatch instead of the instruction helper wrapper. The generated `zr_aot_throw_direct` block validates the payload slot, clears pending control, snapshots the payload `SZrTypeValue`, normalizes it with `ZrCore_Exception_NormalizeThrownValue` and `ZrCore_Exception_NormalizeStatus`, then calls `execution_unwind_exception_to_handler(state, &zr_aot_call_info)`. When the unwind remains inside the same generated frame, it refreshes `frame.callInfo`, `frame.slotBase`, `state->callInfoList`, and `state->stackTop`, computes `zr_aot_next_instruction` from the active program counter, and dispatches through the generated function's switch. If no in-frame handler exists, generated C calls `ZrCore_Exception_Throw(state, state->currentExceptionStatus)` and exits through `ZR_AOT_C_FAIL()`, preserving the generated-function boundary instead of falling back to `ZrLibrary_AotRuntime_Throw`. The C backend no longer emits `ZrLibrary_AotRuntime_Throw` from `backend_aot_c*.c`.

`END_FINALLY` now lowers through direct generated C pending-control and finally-resume logic instead of the instruction helper wrapper. The generated `zr_aot_end_finally_direct` block validates the active generated call frame, pops the matching handler with `execution_find_handler_state` and `execution_pop_exception_handler`, then switches on `state->pendingControl.kind`. `ZR_VM_PENDING_CONTROL_NONE` falls through, pending exceptions resume with `execution_unwind_exception_to_handler`, and pending return/break/continue either resume an outer finally with `execution_resume_pending_via_outer_finally` or jump directly with `execution_jump_to_instruction_offset`. Pending return values copy through `ZrCore_Value_Copy` before the resume path, direct jumps clear pending control with `execution_clear_pending_control`, and all successful non-fallthrough paths refresh the generated frame, compute `zr_aot_next_instruction`, and dispatch through the generated switch. The C backend no longer emits `ZrLibrary_AotRuntime_EndFinally` from `backend_aot_c*.c`; LLVM parity and full generated exception execution fixtures remain future work.

Root/export `FUNCTION_RETURN` now separates module export publication from the return instruction. When the root generated function has exported variables, generated C emits `zr_aot_publish_exports_direct` before the same `zr_aot_direct_return` block used by non-export returns. Export publication is now a generated loop with direct value copy or direct native-closure construction for callable exports; unsupported captured export shapes fail explicitly with `unsupported AOT module export closure capture materialization`. `FUNCTION_RETURN` no longer falls back to `ZrLibrary_AotRuntime_Return(state, &frame, ..., ZR_TRUE)`. The direct return block still performs handler discard, return escape metadata, captured stack-value close, result copy, and caller stack-top restoration.

Global object reads now have a direct generated-C path in `backend_aot_c_lowering_values.c`. `GET_GLOBAL` resolves the destination slot with `ZrCore_Stack_GetValue(frame.slotBase + slot)`, checks `state->global != ZR_NULL` and `state->global->zrObject.type == ZR_VALUE_TYPE_OBJECT`, then copies `state->global->zrObject` with `ZrCore_Value_Copy`; missing or non-object globals reset the destination to null with `ZrCore_Value_ResetAsNull`. Generated modules include `zr_vm_core/global.h` for the direct `state->global->zrObject` access, and the C backend no longer emits `ZrLibrary_AotRuntime_GetGlobal` for `GET_GLOBAL`. LLVM still keeps its helper route until LLVM parity is handled.

Runtime type reflection now also has a direct generated-C path. `TYPEOF` emits `zr_aot_value_exec_typeof`, resolves the destination and source stack values directly from `frame.slotBase`, calls `ZrCore_Reflection_TypeOfValue(state, zr_aot_source, zr_aot_destination)`, and fails through `ZR_AOT_C_FAIL()` when the destination is invalid or reflection fails. Generated modules include `zr_vm_core/reflection.h`, and the C backend no longer emits `ZrLibrary_AotRuntime_TypeOf` for `TYPEOF`. LLVM still keeps its helper route until LLVM parity is handled.

Runtime object/struct conversion opcodes now have an explicit generated-C core semantic boundary. `TO_OBJECT` and `TO_STRUCT` emit `zr_aot_value_exec_to_object` / `zr_aot_value_exec_to_struct`, resolve destination and source stack values directly from `frame.slotBase`, load the type-name operand from `frame.function->constantValueList[index]`, and call `ZrCore_Execution_ToObject(state, frame.callInfo, ...)` or `ZrCore_Execution_ToStruct(state, frame.callInfo, ...)`. Generated modules include `zr_vm_core/execution.h`, and the C backend no longer emits `ZrLibrary_AotRuntime_ToObject` or `ZrLibrary_AotRuntime_ToStruct` for those opcodes. This keeps conversion behavior tied to the core interpreter semantic functions while removing the AOT instruction-helper boundary; full inline struct boxing/unboxing from frame byte spans and LLVM parity remain separate contracts.

The legacy quickened integer arithmetic family now has its own generated-C lowering module in `backend_aot_c_lowering_legacy_int_arithmetic.c`. `ADD_INT`, `ADD_INT_CONST`, `SUB_INT`, and `SUB_INT_CONST` read legacy `SZrTypeValue` stack slots, validate signed/unsigned integer operands, and emit direct C expressions such as `zr_aot_left_int + zr_aot_right_int`, `zr_aot_left_int + zr_aot_right_literal`, `zr_aot_left_int - zr_aot_right_int`, and `zr_aot_left_int - zr_aot_right_literal`. Const forms read the right operand from the function constant table and format signed, unsigned, or bool integer-like constants as `TZrInt64` C literals. Missing or unsupported constants now emit `unsupported AOT ADD_INT_CONST constant` / `unsupported AOT SUB_INT_CONST constant` and fail through `ZR_AOT_C_FAIL()` instead of falling back to `ZrLibrary_AotRuntime_AddIntConst`, `ZrLibrary_AotRuntime_SubInt`, or `ZrLibrary_AotRuntime_SubIntConst`.

Generic numeric arithmetic now has a generated-C direct lowering module in `backend_aot_c_lowering_generic_numeric_arithmetic.c`. Generic `ADD`, `SUB`, `MUL`, `DIV`, `MOD`, and `NEG` route through direct emitters instead of printing `ZrLibrary_AotRuntime_Add`, `Sub`, `Mul`, `Div`, `Mod`, or `Neg` calls. The generated code accepts only numeric `SZrTypeValue` operands for this subset, chooses float, signed integer, or unsigned integer extraction based on the runtime value tags, and emits direct `+`, `-`, `*`, `/`, `%`, `fmod(...)`, and unary `-` C expressions. `DIV` and `MOD` emit explicit zero guards with `divide by zero` and `modulo by zero`; generic `MOD` now follows the interpreter/helper float-fallback shape for numeric operands by emitting `fmod(zr_aot_left_float, zr_aot_right_float)` and writing a `ZR_VALUE_TYPE_DOUBLE` result whenever either operand is float. Unsupported nonnumeric, string, dynamic/meta, or helper-only bool-as-numeric cases report `unsupported AOT generic numeric arithmetic` and exit through `ZR_AOT_C_FAIL()` rather than silently falling back to a VM arithmetic helper. String concatenation and meta arithmetic therefore remain explicit unsupported/dynamic boundaries until typed SemIR/call lowering gives them a non-helper AOT contract.

Generic primitive conversions now have a generated-C direct lowering module in `backend_aot_c_lowering_generic_conversion.c`. Generic `TO_BOOL`, `TO_INT`, `TO_UINT`, and `TO_FLOAT` route through direct emitters instead of printing `ZrLibrary_AotRuntime_ToBool`, `ToInt`, `ToUInt`, or `ToFloat` calls for the primitive subset. The generated code handles null/bool/signed/unsigned/float source tags with direct `ZR_VALUE_FAST_SET` writes, direct scalar casts, or `*zr_aot_destination = *zr_aot_source` for same-category copies. Unsupported non-primitive, object/string, dynamic/meta, and other conversion cases report `unsupported AOT generic primitive conversion` and exit through `ZR_AOT_C_FAIL()` rather than silently falling back to a runtime conversion helper.

Generic primitive truthiness now has a generated-C direct lowering module in `backend_aot_c_lowering_generic_logical.c`. Generic `LOGICAL_NOT` and generic `JUMP_IF` route through the same primitive truthiness writer instead of printing `ZrLibrary_AotRuntime_LogicalNot` or `ZrLibrary_AotRuntime_IsTruthy` calls. The generated code treats null as false, bool as its normalized payload, signed/unsigned integers as `!= 0`, and floats as `!= 0.0`; `LOGICAL_NOT` writes `!zr_aot_truthy` with `ZR_VALUE_FAST_SET`, while `JUMP_IF` emits a direct `if (!zr_aot_truthy) goto ...` branch. String/object/meta/dynamic truthiness remains outside this direct primitive subset and reports `unsupported AOT generic primitive truthiness` through `ZR_AOT_C_FAIL()` rather than silently asking the VM helper for a truth value.

Generic primitive equality now lives in the same logical lowering module. Generic `LOGICAL_EQUAL` and `LOGICAL_NOT_EQUAL` validate the primitive runtime value tags, compare null/bool/signed/unsigned/float values directly, normalize bool payloads before comparison, and write a bool result with `ZR_VALUE_FAST_SET`. String/object/meta/dynamic equality remains outside this direct primitive subset and reports `unsupported AOT generic primitive equality` instead of falling back to `ZrLibrary_AotRuntime_LogicalEqual` or `ZrCore_Value_Equal`.

The bool binary `LOGICAL_AND` and `LOGICAL_OR` bytecode opcodes now also have direct generated-C lowering in `backend_aot_c_lowering_generic_logical.c`. When those opcodes are present, generated C validates both operands are bool values, normalizes `nativeBool` payloads, emits `zr_aot_left_bool && zr_aot_right_bool` or `zr_aot_left_bool || zr_aot_right_bool`, and writes a bool destination directly. Unsupported non-bool operands report `unsupported AOT bool logical binary` and fail through `ZR_AOT_C_FAIL()` rather than calling `ZrLibrary_AotRuntime_LogicalAnd` or `ZrLibrary_AotRuntime_LogicalOr`. Source-level `&&` / `||` still normally lowers to short-circuit branch CFG before `LOGICAL_AND` / `LOGICAL_OR`; the source-level shared-library smoke therefore validates `zr_aot_jump_if_bool_false` / `zr_aot_bool_not_exec` execution and helper absence, while the logical source-contract test locks the bytecode direct-emitter shape.

Typed string equality now has its first direct generated-C lowering in the same logical module. `LOGICAL_EQUAL_STRING` and `LOGICAL_NOT_EQUAL_STRING` validate string operands, cast the payloads with `ZR_CAST_STRING`, read byte lengths and native byte pointers through `ZrCore_String_GetByteLength` / `ZrCore_String_GetNativeString`, compare equal-length byte spans with `memcmp`, and write bool destinations with `ZR_VALUE_FAST_SET`. Unsupported operands or null string objects report `unsupported AOT string equality` and fail through `ZR_AOT_C_FAIL()` rather than calling `ZrLibrary_AotRuntime_LogicalEqualString`, `ZrLibrary_AotRuntime_LogicalNotEqualString`, or `ZrCore_String_Equal`. Generated modules now emit `#include "zr_vm_core/string.h"` and `#include <string.h>` so downstream C compilation sees the string accessors and byte comparison function. LLVM string equality still keeps its existing helper route until LLVM parity is handled.

The generated-frame ABI is still mixed during this migration: inline struct payloads are addressed as byte spans from `frame.slotBase`, while scalar temporaries are still addressed as legacy stack value slots with `ZrCore_Stack_GetValue(frame.slotBase + slot)`. Typed call/return lowering is executable only for the first proven POD direct-call subset; large hidden returns, whole non-POD drop/finalization, mismatched nested inline struct field transfer, generated meta-call execution, and tail-call overlap remain future slices. The current AOT behavior for proven-but-unsupported value fields is an explicit generated failure boundary rather than helper fallback.

Dynamic, generic, and static resolved function calls now have direct generated-C core-call paths for the existing stack-call ABI. `backend_aot_c_function_body.c` finds the AotExecIR row for the current bytecode instruction and treats `ZR_SEMIR_OPCODE_DYN_CALL` / `ZR_SEMIR_OPCODE_DYN_TAIL_CALL` as dynamic even when quickening has rewritten the bytecode opcode to a generic `FUNCTION_CALL` / `FUNCTION_TAIL_CALL` family opcode. Static resolved callsites still win and use `zr_aot_direct_static_function_call`; generated C resolves callee metadata from the callable slot with `ZrCore_Closure_GetMetadataFunctionFromValue`, stages callable and arguments at the caller frame-storage top, prepares the resolved VM call frame with `ZrCore_Function_PreCallPreparedResolvedVmFunctionWithArgumentSource`, invokes the known generated thunk `zr_aot_fn_N`, posts the call through `ZrCore_Function_PostCall`, and refreshes `frame.callInfo` / `frame.slotBase` from the active caller frame. The staged window keeps call setup out of caller inline payload bytes, while the argument-source frame parameters keep byte-backed VALUE arguments copyable from their original layout slots. Dynamic and unresolved generic function calls are emitted by the shared C template in `backend_aot_write_c_core_function_call()`. The template validates frame slots, anchors the callable and destination slots with `ZrCore_Function_StackAnchorInit`, sets `state->stackTop.valuePointer = callBase + 1 + argumentCount`, invokes `ZrCore_Function_CallAndRestoreAnchor`, restores the destination anchor, copies the result with `ZrCore_Value_Copy`, and refreshes the active frame. The C backend no longer emits `ZrLibrary_AotRuntime_Call(state, &frame, ...)` for dynamic call lowering, no longer emits `ZrLibrary_AotRuntime_PrepareDirectCall` for generic direct function calls, and no longer emits `ZrLibrary_AotRuntime_PrepareStaticDirectCall` / `ZrLibrary_AotRuntime_FinishDirectCall` for static resolved direct calls; generated meta-call execution remains a separate boundary.

Typed scalar arithmetic has a first non-helper AOT lowering slice in `backend_aot_c_lowering_typed_arithmetic.c`. The function-body dispatcher routes non-const `ADD_SIGNED`, `ADD_UNSIGNED`, `SUB_SIGNED`, `SUB_UNSIGNED`, `MUL_SIGNED`, `MUL_UNSIGNED`, `DIV_SIGNED`, `DIV_UNSIGNED`, `ADD_FLOAT`, `SUB_FLOAT`, `MUL_FLOAT`, `DIV_FLOAT`, `NEG_SIGNED`, and `NEG_FLOAT` to emitter helpers instead of printing `ZrLibrary_AotRuntime_*` arithmetic calls. Those emitters load the source `SZrTypeValue` slots, validate the signed/unsigned/float category selected by the opcode, and emit direct C scalar expressions such as `zr_aot_left_scalar + zr_aot_right_scalar`, `-`, `*`, `/`, and `-zr_aot_source_scalar` into `ZR_VALUE_FAST_SET`. Division checks zero before the generated C division, reports `divide by zero`, and exits through `ZR_AOT_C_FAIL()` rather than falling through after an error.

The same lowering module now covers const-specialized signed and unsigned `+`, `-`, `*`, and `/` opcode families. It reads the right-hand operand from the function constant table, formats it as a typed C literal, checks the source slot type at runtime, and emits `zr_aot_left_scalar +|-|*|/ zr_aot_right_literal`. If the constant table entry does not match the opcode's signed/unsigned contract, the generated code raises an explicit unsupported typed arithmetic constant error instead of falling back to a VM arithmetic helper.

Signed and unsigned typed `%` now follow the same direct scalar lowering shape for both register/register and register/constant opcode families. `MOD_SIGNED`, `MOD_UNSIGNED`, `MOD_SIGNED_CONST`, and `MOD_UNSIGNED_CONST` emit `zr_aot_left_scalar % zr_aot_right_scalar` or `zr_aot_left_scalar % zr_aot_right_literal`, validate the selected signed/unsigned operand category, and report `modulo by zero` before the generated C `%` expression. The quickened adjacent signed-add plus const-modulo form `ADD_SIGNED_MOD_CONST` now emits `(zr_aot_left_scalar + zr_aot_right_scalar) % zr_aot_mod_literal`, normalizes a negative divisor, and preserves the same modulo-by-zero failure behavior. Typed `MOD_FLOAT` and numeric generic `MOD` float fallback now use direct generated-C `fmod` paths; nonnumeric/meta modulo and LLVM parity remain separate contracts.

Typed signed load-const arithmetic has been split into `backend_aot_c_lowering_typed_arithmetic_load_const.c` so the main typed arithmetic lowering file remains focused. `ADD_SIGNED_LOAD_CONST`, `SUB_SIGNED_LOAD_CONST`, `MUL_SIGNED_LOAD_CONST`, `DIV_SIGNED_LOAD_CONST`, and `MOD_SIGNED_LOAD_CONST` now lower to direct generated C expressions instead of routing through arithmetic runtime helpers. The generated C materializes the right-hand constant into the requested stack slot with `ZR_VALUE_FAST_SET`, reads the left scalar, emits `+`, `-`, `*`, `/`, or `%`, and reports divide-by-zero or modulo-by-zero before evaluating the C operator. The same module now also covers `ADD_SIGNED_LOAD_STACK_CONST`, `SUB_SIGNED_LOAD_STACK_CONST`, `MUL_SIGNED_LOAD_STACK_CONST`, `DIV_SIGNED_LOAD_STACK_CONST`, and `MOD_SIGNED_LOAD_STACK_CONST`: generated C materializes the left source slot into the requested staging slot with direct `SZrTypeValue` assignment, then emits the same scalar C operators and zero guards. `ADD_SIGNED_LOAD_STACK_LOAD_CONST` now materializes both the left source stack value and the right constant staging slot before emitting a direct `+` expression. `ADD_SIGNED_LOAD_STACK` and `MUL_SIGNED_LOAD_STACK` share `backend_aot_c_lowering_typed_arithmetic_load_stack.c` because they consume two already-materialized stack slots; the generated C reads both slots directly, validates signed integer categories, emits `zr_aot_left_scalar + zr_aot_right_scalar` or `zr_aot_left_scalar * zr_aot_right_scalar`, preserves the add path's left-type result tag, and stores multiply as `ZR_VALUE_TYPE_INT64`. The executable-subset gate accepts these signed quickening forms so they can participate in generated AOT C modules.

Typed integer bitwise and shift operations have their own AOT lowering module in `backend_aot_c_lowering_typed_bitwise.c`. `BITWISE_NOT`, `BITWISE_AND`, `BITWISE_OR`, `BITWISE_XOR`, `SHIFT_LEFT_INT`, `SHIFT_RIGHT_INT`, `BITWISE_SHIFT_LEFT`, and `BITWISE_SHIFT_RIGHT` now validate integer operands and emit C `~`, `&`, `|`, `^`, `<<`, and `>>` expressions instead of direct `ZrLibrary_AotRuntime_Bitwise*` / `Shift*Int` helper calls. Left shifts are generated through an unsigned intermediate before casting back to `TZrInt64`, and `BITWISE_SHIFT_RIGHT` preserves the existing logical-right-shift contract by using an unsigned left operand. Shift counts are checked as `[0, 63]` before generating the shift expression so undefined C shift counts do not become silent AOT behavior.

The same bitwise lowering module now covers the integer-primitive subset for generic `SHIFT_LEFT` and `SHIFT_RIGHT` bytecodes in generated AOT C. The function-body dispatcher routes those opcodes to direct emitters instead of printing `ZrLibrary_AotRuntime_ShiftLeft` or `ZrLibrary_AotRuntime_ShiftRight`. Generated C validates both operands with `ZR_VALUE_IS_TYPE_INT`, extracts signed or unsigned integer payloads through the existing integer-like helpers, checks shift counts as `[0, 63]`, writes `ZR_VALUE_TYPE_INT64` results directly, and reports `unsupported AOT generic integer shift` for non-integer/meta/dynamic operands. Generic left shift uses the unsigned intermediate `zr_aot_left_unsigned << zr_aot_shift_count`; generic right shift uses signed `zr_aot_left_scalar >> zr_aot_shift_count`.

Typed float modulo has a focused generated-C lowering module in `backend_aot_c_lowering_typed_float_mod.c`. `MOD_FLOAT` now validates float-like operands with `ZR_VALUE_IS_TYPE_FLOAT`, reads `nativeDouble` payloads, checks `zr_aot_right_scalar == 0.0` before evaluating the modulo, reports `modulo by zero`, and writes the `ZR_VALUE_TYPE_DOUBLE` result with `ZR_VALUE_FAST_SET(... fmod(zr_aot_left_scalar, zr_aot_right_scalar) ...)`. The function-body dispatcher routes `MOD_FLOAT` through this emitter instead of printing `ZrLibrary_AotRuntime_ModFloat(state, &frame, ...)`, and generated modules emit `#include <math.h>` for the direct `fmod` call. Numeric generic `MOD` now uses the same generated-C math include for its float fallback; LLVM parity remains a separate future contract.

Typed power now has a focused generated-C lowering module in `backend_aot_c_lowering_typed_power.c`. `POW_SIGNED` and `POW_UNSIGNED` validate signed/unsigned integer operands, guard unsupported power domains with `power domain error`, emit exponentiation-by-squaring loops with `zr_aot_power_result *= zr_aot_power_base` and `zr_aot_power_exponent >>= 1`, and write `ZR_VALUE_TYPE_INT64` or `ZR_VALUE_TYPE_UINT64` results directly. `POW_FLOAT` validates float operands, emits `pow(zr_aot_left_scalar, zr_aot_right_scalar)`, and writes `ZR_VALUE_TYPE_DOUBLE`. The function-body dispatcher routes all three typed power opcodes through direct emitters instead of printing typed power runtime helper calls.

Generic `POW` now has a separate generated-C meta-boundary lowering module in `backend_aot_c_lowering_generic_power.c`. This opcode is not primitive numeric power in the current VM; the interpreter and previous AOT helper look for `ZR_META_POW` and otherwise write `null`. Generated AOT C now emits that no-meta/null behavior directly with `ZrCore_Value_GetMeta(state, zr_aot_left, ZR_META_POW)` and `ZrCore_Value_ResetAsNull(zr_aot_destination)`. If a real meta power function is present, generated C reports `unsupported AOT generic power meta dispatch` and exits through `ZR_AOT_C_FAIL()` instead of silently calling `ZrLibrary_AotRuntime_Pow`. Full generated meta-call execution, source-level `**` parser support, and LLVM parity remain separate future contracts.

The typed numeric comparison families now use the same lowering module. Signed, unsigned, and float `==`, `!=`, `>`, `<`, `>=`, and `<=` opcodes validate their operand category and emit direct C comparison expressions such as `zr_aot_left_scalar == zr_aot_right_scalar` into a boolean destination. This removes the direct `ZrLibrary_AotRuntime_Logical*` helper path for those typed numeric comparison opcodes while leaving object/meta/dynamic equality and LLVM parity on explicit future/runtime contract paths.

Typed bool logical operations have a separate narrow lowering module in `backend_aot_c_lowering_typed_logical.c`. `LOGICAL_EQUAL_BOOL` and `LOGICAL_NOT_EQUAL_BOOL` validate that both operands are bool values, normalize their stored `nativeBool` payloads, and emit `zr_aot_left_bool == zr_aot_right_bool` or `zr_aot_left_bool != zr_aot_right_bool` into a boolean destination. Statically bool unary `!` now compiles to `LOGICAL_NOT_BOOL`, which validates a bool source and emits `!zr_aot_source_bool`. The function-body dispatcher routes these typed bool opcodes through direct emitters instead of printing `ZrLibrary_AotRuntime_LogicalEqualBool` / `LogicalNotEqualBool` / `LogicalNot` calls for the proven-bool cases.

Typed bool control flow now has a non-truthiness lowering path. `compiler_create_jump_if_false_for_condition()` emits `JUMP_IF_BOOL_FALSE` when the condition expression is statically bool, while preserving generic `JUMP_IF` for dynamic truthiness. The interpreter validates a bool frame slot and branches on `nativeBool` directly. AOT C lowers this opcode in `backend_aot_c_lowering_control.c` by reading the bool slot instead of calling `ZrLibrary_AotRuntime_IsTruthy`; when the condition slot has declared bool scalar-local coverage, it synchronizes the frame slot into `zr_aot_bN` and emits `if (!zr_aot_bN) { goto ...; }`, with `zr_aot_condition_bool` retained as the fallback predicate for unproven slots. Signed numeric compare-plus-false-branch fusion now treats `JUMP_IF_BOOL_FALSE` as the same false-branch consumer as generic `JUMP_IF`, so statically typed `>`, `>=`, and `==` conditions can still collapse into `JUMP_IF_LESS_EQUAL_SIGNED`, `JUMP_IF_GREATER_SIGNED`, `JUMP_IF_NOT_EQUAL_SIGNED`, or `JUMP_IF_NOT_EQUAL_SIGNED_CONST` instead of materializing a bool temporary. AOT C lowers those fused signed branch opcodes by reading the two signed slots or a signed constant and emitting direct C comparisons such as `zr_aot_left_scalar <= zr_aot_right_scalar` or `zr_aot_left_scalar != zr_aot_right_literal` before a generated `goto`; it no longer calls `ZrLibrary_AotRuntime_ShouldJumpIfGreaterSigned` for the proven signed-branch family. When both fused signed branch operands have declared `i64` scalar-local coverage, the branch helper now synchronizes the frame-slot operands directly into `zr_aot_sN` locals from `zr_aot_left->value.nativeObject.nativeInt64` / `zr_aot_right->value.nativeObject.nativeInt64` and emits the branch predicate from those locals, for example `if (zr_aot_s2 <= zr_aot_s4) {`. The focused signed const branch path follows the same left-operand rule: when `JUMP_IF_NOT_EQUAL_SIGNED_CONST` has proven `i64` scalar-local coverage for the left slot, generated C syncs that frame slot into `zr_aot_sN` and emits `if (zr_aot_sN != zr_aot_right_literal) {` instead of branching on `zr_aot_left_scalar`.

AOT C scalar-local declarations now merge source typed-local evidence and SemIR destination evidence as per-slot kind flags rather than treating the slot kind as mutually exclusive. This lets a reused slot declare multiple C mirrors, for example both `zr_aot_s17` and `zr_aot_f17`, without losing either lifetime's static type evidence. The focused M2 scalar backend uses that to write signed `i64` shift and bit-not temporary destinations directly to `sN` locals when SemIR proves the destination type, such as `zr_aot_s17 = (TZrInt64)((TZrUInt64)zr_aot_s13 << zr_aot_s1);`, `zr_aot_s18 = zr_aot_s14 >> zr_aot_s1;`, and `zr_aot_s17 = ~zr_aot_s1;`, while still mirroring the result back to the existing frame slot ABI.

Typed numeric conversions now have a first direct scalar lowering slice. Explicit `<float>` casts from statically signed integer sources compile to `TO_FLOAT_SIGNED`, explicit `<float>` casts from statically unsigned integer sources compile to `TO_FLOAT_UNSIGNED`, explicit `<int>` casts from statically float sources compile to `TO_INT_FLOAT`, explicit `<int>` casts from statically unsigned integer sources compile to `TO_INT_UNSIGNED`, explicit `<uint>` casts from statically float sources compile to `TO_UINT_FLOAT`, and explicit `<uint>` casts from statically signed integer sources compile to `TO_UINT_SIGNED`; same-type casts still collapse to the source slot and dynamic/other conversions keep the generic `TO_FLOAT` / `TO_INT` / `TO_UINT` paths. The interpreter reads the source through generated-frame slots and stores direct scalar results into `SZrTypeValue` destinations. AOT C lowers these opcodes in `backend_aot_c_lowering_typed_conversion.c` with `zr_aot_convert_signed_to_float`, `zr_aot_convert_unsigned_to_float`, `zr_aot_convert_float_to_signed`, `zr_aot_convert_unsigned_to_signed`, `zr_aot_convert_float_to_unsigned`, and `zr_aot_convert_signed_to_unsigned` blocks, using `(TZrFloat64)zr_aot_source_scalar`, `(TZrInt64)zr_aot_source_scalar`, `(TZrUInt64)zr_aot_source_scalar`, and an explicit `zr_aot_unsigned_to_signed_limit` formula instead of runtime conversion helpers for these proven numeric cases.

Runtime string conversion now has a direct generated-C core conversion path. `TO_STRING` emits `zr_aot_value_exec_to_string`, resolves source and destination values from `frame.slotBase`, calls `ZrCore_Value_ConvertToString(state, zr_aot_source)`, refreshes `frame.callInfo`, `frame.slotBase`, and `state->stackTop` from the active call info after possible meta conversion work, and writes string/null directly with `ZrCore_Value_InitAsRawObject` or `ZrCore_Value_ResetAsNull` instead of `ZrLibrary_AotRuntime_ToString`. This remains a core semantic boundary because `TO_STRING` may invoke meta conversion; LLVM remains helper-backed until LLVM parity is handled.

This slice deliberately leaves generic/dynamic arithmetic outside the supported primitive subsets, string concatenation, bool-as-numeric helper compatibility for generic numeric arithmetic, full generic power meta-call execution, source-level `**` parsing, dynamic/meta shift behavior outside the integer-primitive generic shift subset, bool/string/object conversions, object/meta/dynamic equality, generic equality outside the primitive subset, string/object/meta/dynamic truthiness, and logical `&&`/`||` on explicit unsupported or existing helper paths. The interpreter's historical float-fallback behavior for `ADD_INT` / `SUB_INT` is also not treated as an AOT helper escape hatch; generated C now covers the integer subset directly and fails explicitly outside that subset. The remaining dynamic paths still carry VM semantics that need separate metadata or constant-materialization contracts before they can be converted to direct C expressions without silently changing behavior.

Remaining dynamic and meta SemIR opcodes keep explicit runtime or unsupported-boundary behavior until their static metadata contracts are proven. Value-type SemIR is therefore a typed-layout contract first; the interpreter is being moved onto the same layout facts in incremental source-level slices.

## Interpreter Boundary

The active interpreter now has a first source-level local `struct` value-type execution slice for primitive POD fields:

- `ZrCore_Function_ResolvePrototypeFrameFieldLayout()` resolves a struct field name to byte offset, byte size, primitive value kind, nested type id, or embedded `SZrTypeValue` marker from compiled prototype metadata.
- `execution_inline_frame_try_get_member*()` and `execution_inline_frame_try_set_member*()` resolve known inline struct receivers through `SZrFunctionFrameSlotLayout`, then load/store the primitive field directly from the frame byte span.
- `execution_inline_frame_try_copy_stack_slot()` copies inline-struct slots by layout, and can materialize a boxed/source constructor object into an inline destination by writing primitive fields at their layout offsets.
- The interpreter dispatch checks these inline-frame helpers before falling back to object/array/string member dispatch for `GET_MEMBER`, `SET_MEMBER`, `GET_MEMBER_SLOT`, and `SET_MEMBER_SLOT`.
- Quickening preserves by-value local copy intent by forwarding `GET_STACK p -> temp; SET_STACK q <- temp` into a direct inline-struct `SET_STACK q <- p` when both slots share the same inline type layout.
- Source-level by-value struct arguments now preserve inline layout facts for the caller staging slot and the callee parameter slot. When a POD struct parameter is mutated inside the callee, field stores target the callee frame byte span and do not alias the caller's source payload.
- Runtime prototype field-layout resolution can recover the entry function's prototype metadata from the active call stack when compiled child functions no longer retain owner links after quickening/compilation cleanup.
- Source-level POD struct returns now preserve inline layout facts for the caller result slot from resolved call return types. The runtime return path copies the callee inline return payload into the caller inline target span, so mutating a returned value does not alias another inline payload.
- Inline stack slot hint floors keep struct materialization, argument, and return slots from being trimmed and reused as scalar scratch while a layout-aware expression is still active.
- Primary-expression compilation tracks and returns the actual result slot. Normalizing a call expression into a requested target slot therefore copies from the known result slot instead of assuming `stackSlotCount - 1`, which is no longer reliable once inline payload slots are preserved above scalar temporaries.
- Typed bool equality, typed bool logical-not, typed signed/float negation, `JUMP_IF`, and `JUMP_IF_BOOL_FALSE` condition dispatch now read and write through frame metadata slots instead of the old flat stack mirror. This keeps scalar bool and numeric temporaries aligned with the same generated-frame layout used by inline value payloads.
- Typed signed/unsigned-to-float, float-to-signed/unsigned, and signed/unsigned integer cross-cast opcodes read and write through the same generated-frame slot accessors, keeping explicit numeric cast temporaries aligned with the inline-frame ABI. `TO_INT_UNSIGNED` uses an explicit high-bit wrap formula so `uint` values above `INT64_MAX` do not rely on implementation-defined C signed casts.

This slice intentionally stays narrow: it covers source-level local struct construction/materialization, local by-value copy, POD struct by-value parameter isolation, POD struct by-value return isolation, and primitive field get/set. Large/non-trivial hidden returns, managed/reference fields, tail-call overlap stress, and hidden-return/AOT executable body replacement remain later milestones.
