# 02 · `using` 使用场景全集 + 按需插件加载守卫

## 2. 问题：`%import` 今天不安全也不"按需"

现状（[00-current-state.md](00-current-state.md) §4）：`var x = %import("foo")` 立即解析、无失败降级、无"加载成功才让类型可见"的语义。对**外部 DLL 插件**（可能未安装、版本不匹配、签名漂移）这是危险的——块内代码会假设类型存在。

## 3. 目标语法：`using` 守卫按需插件

```zr
// 按需加载一个外部插件 DLL；只有当 "render.vulkan" 已被有效加载
// 且其导出的类型签名与本模块编译期记录的签名一致时，才进入 block。
using (var [gpu] = %import("render.vulkan")) {
    // 块内：gpu.Device / gpu.Buffer 等类型是“局部类型安全”的——
    // 编译器在此块内按 import 的签名 token 解析它们（见 03 篇）。
    var dev = new gpu.Device();
    using (var [buf] = Unique<gpu.Buffer>(dev.alloc(1024))) {
        buf.upload(data);
    }   // buf 确定性 drop
} else {
    // 可选 else：插件不可用 / 签名不匹配时的降级路径。
    log.warn("vulkan backend unavailable, falling back");
    useSoftwareRenderer();
}
```

要点：

1. **按需**：`%import` 在 `using` 守卫位置时，语义从"导入即解析"变为"尝试加载，返回加载结果"。按需检查所有使用的token签名是否一致。此时 `%import`返回 `DynamicModule<T>`，默认不带 `else`分支则走严格匹配签名，如有 `else`分支，则默认走该分支
2. **签名守卫**：进入 block 的条件不是"指针非空"，而是"插件导出的类型签名 token 与本模块编译期记录的引用签名**一致**"（见 [03-metadata-and-token-model.md](03-metadata-and-token-model.md)）。
3. **局部类型安全**：块内对 `gpu.*` 的成员访问、构造、cast，编译器按 import 签名静态解析；块外这些类型不可见。
4. **降级**：可选 `else` 分支处理"未加载 / 签名不匹配" 类比"特性探测"。无 `else` 时，加载失败则跳过整个 block（抛出MethodSignatureMismatchError()。此外 `else`也能在处理所有权的时候，处理已被剥夺或者回收的情况，同样在默认没有有效所有权时，它会抛出异常例如NullReferenceError。

## 4. 加载语义与生命周期

```
%import("plugin") 在 using 守卫位置的求值：
  1. 查 native registry（native_registry.h: FindModule / EnsureProjectDescriptorPlugin）
       命中已加载？ → 取已加载实例
       未加载？     → 尝试加载（FFI: ZrFfi_LoadLibrary / descriptor plugin）
  2. 加载失败 → 守卫求值为 “unavailable” → 跳过 block / 走 else
  3. 加载成功 → 校验导出签名 token vs 本模块引用签名
       不匹配 → “unavailable” → else
       匹配   → 守卫求值为 “available(handle)” → 绑定 p，进入 block
  4. block 退出：
       默认 —— 释放本作用域对插件的引用计数（插件可被卸载或缓存，由 registry 决定）
       可加修饰要求常驻（见下）
```

加载结果天然是一个 **union**：`DynamicModule<T> where T: zr.Module { Unavailable; @Available(m: Module); }`。这把第 04 篇的 union 与本篇的守卫统一起来——`using (var [p]: DynamicModule<Plugin>.Available = %import(...))` 本质是对 `@Available` variant 的解构守卫。当前 typed 形态已接入 guard lowering：`using (var [p]: DynamicModule<Plugin>.Available = %import(...))` 与 `using (var [p]: DynamicModule<Plugin> = %import(...))` 都会按唯一 `@Available` 展开；裸 `Module` 在类型解析中按内建类型处理；无类型注解的 `using (var [p] = %import(...))` 也会在 pattern guard resource 为 `%import` 时推断为当前脚本 `DynamicModule` 的唯一 `@Available` 默认 variant，并复用同一 guard helper。`PluginLoad.Available` 已作为内置注解 surface 接入同一路径，因此 `using (var [p]: PluginLoad.Available = %import(...))` 不要求脚本预先声明 `DynamicModule` union，也会把成功分支 module handle 绑定为 payload `p`。typed、no-annotation 与 `PluginLoad.Available` payload binding 会在进入 block 前交给插件 guard escape scanner，因此未 `share()` 的 `p` 不能经 `return`、`throw`、`out`、闭包捕获、调用实参、构造函数实参、对象字段、数组元素或 generator 延迟输出逃出 guard；这些路径位于 try/catch/finally、generator/out 或带表达式 break/continue 中时也同样被拒绝。如果内层 `if` / block / loop 中把 `p` 赋给外层局部，该外层局部也会继续被当作 guard-scoped 插件别名。

当前 `.share()` 基础提升已能把 guard-scoped plain module handle 显式提升为 shared owner：`var handle = p.share()` 下沉到 `ZrCore_Ownership_NativeSharePlain`，结果可经 `%release(handle)` 释放，同时原 guard payload `p` 保持 plain module 值供 guard 内继续访问。默认 scoped 引用释放基础也已接入：guard 命中后会生成一个隐藏 shared owner，并登记当前 scope cleanup，在 block 退出或既有控制流 cleanup 前发出 `OWN_RELEASE`。该 owner release 现在会通过 core ownership observer 更新 native registry 的 `ownerRefCount`；当最后一个 shared owner 释放后，`ZrLibrary_NativeRegistry_GetModuleRefCount(global, name)` 可观察到计数归零。后续还需要基于该计数补 descriptor DLL safe unload、cached module/native closure invalidation 和热重载策略。

## 5. 与 native registry / FFI 的对接

复用既有基建（[00-current-state.md](00-current-state.md) §4），无需新运行时加载器：

| 步骤                   | 复用的现有 API                                                |
| ---------------------- | ------------------------------------------------------------- |
| 查已加载模块           | `ZrLibrary_NativeRegistry_FindModule`                       |
| 按需加载项目描述符插件 | `ZrLibrary_NativeRegistry_EnsureProjectDescriptorPlugin`    |
| 加载裸 DLL / 取符号    | `ZrFfi_LoadLibrary` / `ZrFfi_Library_GetSymbol`           |
| 失效缓存（热重载）     | `ZrLibrary_NativeRegistry_InvalidateDescriptorPluginSource` |

新增的是**编译期签名记录**与**运行期守卫求值**：编译块时记录"本块依赖 plugin M 的哪些类型/成员签名 token"，运行期守卫在 import 成功后做签名比对。当前已完成前置依赖事实、函数级 token 外壳和运行期签名确认切片：`using` 插件 guard 的 body 会把 `p.*` 访问登记为 canonical `moduleEntryEffects`，并把可签名的 import member effect 写成函数级 `ASSEMBLY_REF` + `TYPE_REF` + `MEMBER_REF` + `SIGNATURE` 前置链；exported callable summary 的 effects 会在确定 `callableChildIndex` 后复制到对应 child function；`MEMBER_REF` 在能解析 provider export 时会附带目标 `METHOD_SIG` / `FIELD_SIG` 子签名和 provider `targetMetadataToken` / `targetSignatureToken` / `targetSignatureHash`，native guarded import 也会通过 native-only fallback 带上 target identity；guard 专用 import native helper 会把实际 caller function 上的 `targetSignatureHash` 与 provider typed export `signatureHash` 比对，并在有目标子签名时确认 caller `MEMBER_REF` 子 blob 与 provider signature blob 字节一致，不匹配或 provider 不可用则返回 unavailable。普通 required import 复用同一校验路径，但 mismatch 直接抛出 `import signature mismatch`。完整的模块级 ref 表、native/source/binary ref→def 绑定、版本/hash 诊断仍在 P0 后续。

## 6. 常驻 vs 作用域引用

默认 block 退出释放对插件的引用。若需要把插件句柄带出作用域（长期持有），用 owner 泛型显式表达：

```zr
var gpuHandle: Shared<PluginModule>;
using (var gpu = %import("render.vulkan")) {
    gpuHandle = gpu.share();   // 提升为 Shared，逃出块作用域
}
// 块外 gpuHandle 仍持有插件引用；插件保持加载直到最后一个 Shared 释放
```

这把"插件生命周期"与第 01 篇的所有权模型统一：插件句柄就是一个 owner 值，引用计数决定 DLL 何时可卸载。

## 8. 开放问题

- 签名不匹配是"静默降级"还是"可选严格模式报错"？抛出MethodSignatureMismatchError()
- 同一插件被多个 `using` 块加载，引用计数与卸载时机？`using` / `.share()` 的 shared owner 现在会增减 registry `ownerRefCount`；真正卸载还要等待 descriptor DLL safe unload 与缓存失效策略。
- 块内拿到的插件类型能否逃逸到块外而不经 `share()`？编译期禁止：当前已基础拒绝 `return p`、`throw p`、`out p`、`return p.callableMember`、typed/untyped `DynamicModule<T>.@Available` payload 和 `PluginLoad.Available` payload 的 `return m` / `throw m` / `out m`、调用实参或构造函数实参传出、对象/数组/模板字符串插值这类持久化容器、decorator metadata、parameter default metadata、signature type metadata、foreach binding type metadata、generator 延迟输出和闭包/嵌套函数捕获；try/catch/finally、generator/out、declaration/member decorators、函数/方法/meta function parameter default/signature type、foreach binding type 和带表达式 break/continue 中的同类逃逸路径也会被扫描；普通成员值和调用结果按其结果类型处理。更完整跨 region 的分析继续复用借用逃逸模型推进。
