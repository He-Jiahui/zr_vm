# 04 · `union` 关键字：带字段的代数数据类型

## 1. 目标语法

```zr
// 基本：variant 可无字段、带位置字段、带具名字段
// using {width, height} = shape.Rect;
// using {w: width, h:height} = shape.Rect; 具名字段别名
// using {width, h:height}; 混合别名与默认名
union Shape {
    Empty;                                  // 无字段
    Circle(radius: float);                  // 位置字段
    Rect { width: float; height: float; }   // 具名字段
}

// using [o] = optional.Some; 元组展开
// 泛型 union —— 与 01 篇所有权泛型、03 篇签名同构
union Option<T> {
    None;
    Some(value: T);
}

union Result<T, E> {
    Ok(value: T);
    Err(error: E);
}

// 与所有权组合：variant 字段本身可以是 owner
union Resource {
    Closed;
    Open(handle: Unique<FileHandle>);
}

// 默认using检查 可以简写using(var [m]:DynamicModule<Plugins>.Available = %import("zr.plugins")){...}
// @在union只能存在一个,作为校验默认值, using 只有在作为默认校验值的时候可以直接在匹配的时候自动展开
// 展开规则 位置字段则变为元组展开, 具名字段必须要求是字段展开 var {width, height}: Shape.Rect
union DynamicModule<T> where T:zr.Module{
    Unavailable;
    @Available(m: Module);
}
```

## 3. 分支：`switch` 模式匹配（穷尽性检查）

扩展现有 `switch`（[zr_language_specification.md](../../zr_language_specification.md) §2.10）以支持 variant 模式与解构绑定：

```zr
switch (s) {
    (Shape.Empty) {
        return 0.0;
    }
    (Shape.Circle(r)) {                       // 解构出 radius 绑定到 r
        return 3.14159 * r * r;
    }
    (Shape.Rect { w: width, h: height }) {    // 具名解构  也可以直接Shape.Rect{width, height} 则使用默认名
```

- **穷尽性检查**：覆盖所有 variant 则无需 default；否则要么补齐，要么写 `()` default。
- **绑定作用域**：解构出的绑定只在该 case 块内可见。
- variant 字段若是 owner（`Unique<...>`），解构按"借用还是移动"由模式决定（默认借用，`move` 修饰移动）。
- **当前落地状态**：已支持无解构的 variant tag case（如 `(Shape.Circle)`）编译为 `__zr_unionVariant` 字符串比较；tuple-style payload 绑定（如 `(Shape.Circle(r))`）和 struct-style payload 绑定（如 `(Shape.Rect { width: w, height: h })`）会读取 `__zr_unionPayloadN` 并创建 case 局部变量。当 `switch` 目标能推断为已知 union 类型时，case 也可省略 union 前缀写作 `(Circle(r))` / `(Empty)`，compiler 会用 subject 类型解析 variant 并纳入穷尽性检查；该补全只作用于 identifier / primary-expression case，不复用 using 专用的裸 `[x]` / `{field}` 默认展开语法。compiler 会拒绝 tuple variant 的 object pattern（如 `Shape.Circle { radius: r }`）和 struct variant 的 tuple pattern（如 `Shape.Rect(w, h)`），保持位置字段只能 tuple 解构、具名字段必须字段解构。payload 绑定会按匹配到的 variant 字段声明注册类型；`Unique<T>` / `Shared<T>` / `Loan<T>` owner payload 默认先通过 `OWN_BORROW` 绑定为 `Borrow<T>`，因此默认解构出的 `handle` 不能直接 `%release(handle)`，`Weak<T>` 保持 weak 语义。显式 `move` 可写在 switch payload 本地 binding 前：tuple 形态如 `(Open(move handle))`，struct 形态如 `(Open { handle: move h })` 或同名 `(Open { move handle })`；struct switch pattern 方向仍是 `field: local`。typed union 本地标识符作为 `switch` 目标、声明 initializer、简单赋值右值或构造器赋值目标时会保留/使用 inline local slot；member 表达式 `switch (holder.choice)` 也会保留编译出的 inline union subject slot，避免 tag/payload 临时值覆盖 subject。runtime inline-frame `GET_MEMBER` 和 `SET_STACK` 可直接处理 inline tag/payload bytes；carrier object 路径仍作为构造器桥和非 inline fallback。无 default 的 union `switch` 会检查所有 variants 是否被覆盖；缺失 variant 时编译报错，写 `()` default 可显式兜底；重复覆盖同一 variant 的第二个 case 会作为不可达 case 报错，避免重复 case 掩盖真实覆盖状态。
- **本阶段补充**：tuple/struct pattern 的字段数量不匹配现在会输出结构化 `pattern_arity_mismatch`，包含 expected/actual 计数和可执行修复建议；跨 union 的 using variant annotation 错配会输出结构化 `pattern_variant_mismatch`，说明标注的 union/variant 与 resource union 不一致。`pattern_shape_mismatch`、`pattern_unknown_field`、`pattern_arity_mismatch` 与 `pattern_variant_mismatch` 均已有 LSP golden。块式 `using` tuple/object 与 `switch` tuple/struct owner payload 均已支持显式 `move`，默认 borrow 语义保持不变。

## 4. 分支：`using` 单 variant 守卫

把 `using` 的"解构守卫"角色（[02-using-scopes-and-plugin-guards.md](02-using-scopes-and-plugin-guards.md) 角色 D）用于只关心一个 variant 的场景——这也是为什么插件加载 `PluginLoad = Available | Unavailable` 能用 `using` 表达：

```zr
// 仅当 o 是 Some 时进入块，并把内层值绑定为 x；否则走 else
using (var [x]:Optional<T>.Some = optional.Some) {
    print(x);
} else {
    print("empty");
}

// 与所有权统一：Open variant 解构出的 owner 在块退出时确定性 drop
using (var [handle]:Resource.Open = resource.Open) {
    handle.read();
}   // handle drop

// 节省类型声明的另类写法
using (var [handle] = resource.Open){}

// 具名字段必须用字段解构；alias 写作 local: field
using (var {width, h: height}: Shape.Rect = s) {
    print(width);
    print(h);
} else {
    print("not a rect");
}

// 插件守卫即 union 守卫的特例
using (var [gpu] = %import("render.vulkan")) { ... } else { ... }
//      ^ 等价于对 DynamicModule<RenderVulkan>.Available 的 using 解构
```

`using` 解构 = "匹配单个 variant 则进入并绑定 + 作用域析构"，`switch` = "多路穷尽分支"。两者共用同一套模式匹配与解构 lowering。

**当前落地状态**：基础形式 `using (var [r]: Shape.Circle = shape) { ... } else { ... }`、unit variant 空解构 `using (var []: Shape.Empty = shape) { ... } else { ... }` 与 `using (var {width, h: height}: Shape.Rect = shape) { ... } else { ... }` 已接入 parser 与 compiler；variant 由 `guardTypeInfo` 标注，运行时先比较 resource 的 `__zr_unionVariant`，命中后按 tuple 位置或 struct 声明字段顺序读取 `__zr_unionPayloadN` 并注册块内局部绑定，未命中进入可选 `else`。具名 alias 方向固定为 `local: field`，如 `{h: height}` 从 `height` payload 绑定本地 `h`。compiler 会拒绝 `using (var {r: radius}: Shape.Circle = s)` 和 `using (var [w, h]: Shape.Rect = s)` 这类 shape mismatch；唯一 `@` 默认 variant 的省略 annotation 解构也走同一规则。该 shape mismatch 现在通过结构化 `pattern_shape_mismatch` 诊断进入 LSP，message/cause/suggestion 会说明 positional payload 应使用 tuple destructuring、named fields 应使用 object destructuring。旧式 `using (var Variant(x) = value)` 不再作为当前 `using` surface 接受，编译器会提示迁移到 annotation + destructuring；`switch` case pattern 的 `Variant(x)` 语法不受影响。绑定变量会继承 variant 字段声明的类型；`Unique<T>` / `Shared<T>` / `Loan<T>` owner payload 默认先降为 `Borrow<T>` 局部绑定，并通过 `OWN_BORROW` 保持运行时别名语义，因此 guard 块内不能对默认绑定执行 `%release(handle)` 这类 owner-consuming builtin。显式 `move` binding 会保留字段声明的 owner 类型、跳过 `OWN_BORROW`，并在成功绑定后清空 matched payload。当 resource 是 typed union inline local 或 struct 字段里的 inline union 时，`GET_MEMBER`/`GET_MEMBER_SLOT` 读取的 tag/payload 已来自 inline frame bytes；carrier object 路径仍可服务未 materialize 的构造器桥。`@Available` 默认 variant 标记已能解析并做唯一性校验，且已知 union resource 省略 variant 时会按唯一 `@` 自动展开。typed `%import` guard 已支持 `using (var [p]: DynamicModule<Plugin>.Available = %import(...))` 和 `using (var [p]: DynamicModule<Plugin> = %import(...))`，两者都会复用 guard 专用 import helper，命中时把模块对象绑定为 `@Available` payload，失败或签名不匹配进入 `else`；裸 `Module` 类型已作为隐式内建类型通过类型解析。无类型注解的 `using (var [p] = %import("plugin")) { ... } else { ... }` 也会在 pattern guard resource 为 `%import` 时按当前脚本 `DynamicModule` 的唯一 `@Available` 默认 variant 推断并复用 guard helper。typed/untyped import guard payload binding 已复用插件 guard 逃逸 scanner。

**本阶段补充**：当前稳定 surface 包含块式 guard：`using (var [x]: Union.TupleVariant = value) { ... }`、`using (var {local: field}: Union.StructVariant = value) { ... }`、唯一 `@` 默认 variant 的省略 annotation 写法，以及非 import union resource 的无块解构 shorthand：`using [x] = value;`、`using [x]: Union.Variant = value;`、`using {local: field} = value;`。无块 shorthand 的 binding 留在当前作用域且不支持 `else`；no-block `%import` guard 生命周期仍未扩展。数量不匹配会输出结构化 `pattern_arity_mismatch`；标注 union 与 resource union 不一致会输出结构化 `pattern_variant_mismatch`；owner payload 默认 borrow 的运行时生命周期已覆盖普通参数调用窗口和多 payload 布局容量边界。LSP 语义层已同步注册这些 `using`/`switch` payload locals 和闭型泛型 payload 类型。

**显式 move 状态**：块式 `using` 的 tuple/object destructuring 现在可在局部绑定名前加 `move`，例如 `using (var [move handle]: Resource.Open = resource) { ... }`、`using (var {move handle}: Resource.Open = resource) { ... }`、`using (var {move local: field}: Resource.Open = resource) { ... }`。`switch` case pattern 也可对 payload binding 使用同一语义：tuple 写 `(Open(move handle))`，struct 写 `(Open { handle: move h })` 或 `(Open { move handle })`。该标记只改变对应 payload binding：默认 owner payload 仍借用为 `Borrow<T>`，显式 `move` 则保留字段声明的 owner 类型，跳过 `OWN_BORROW`，并在成功绑定后把 matched union payload slot 写成 null。未标记的 sibling payload 继续按默认规则处理。

## 5. 运行时表示

判别式 + payload 的 tagged 布局，复用值类型 / inline layout 基建（参考 [inline-type-layout-and-byte-stack.md](../../core-runtime/inline-type-layout-and-byte-stack.md)）：

```
union 值布局：
  [ tag: 判别式整数 ][ payload: 各 variant 字段的最大尺寸联合 (按 inline layout 对齐) ]
```

- `tag` 选最小够用整型（variant 数 ≤ 256 用 1B）。
- payload 大小 = 各 variant 字段布局的 max；含 owner 字段时，drop 按当前 tag 选择对应 variant 的析构链（接入 [01 篇](01-ownership-as-generics.md) 的确定性 drop）。
- 序列化：在 `SZrCompiledPrototypeInfo`（[constant_reference.h](../../../zr_vm_core/include/zr_vm_core/constant_reference.h)）新增 union 种类，variant 表记录每个 variant 的 tag 值与字段布局；签名层用 [03 篇](03-metadata-and-token-model.md) 的 `UNION` 节点编码。
- **当前落地状态**：signature blob 的 `UNION` 节点已接入 typed export metadata；已知 union 类型会写入 union 基名和泛型实参签名。prototypeData 已能序列化 union prototype 与 variant member，并通过成员级 metadata 常量记录 variant tag、kind、payload 字段数、字段声明名、类型名、运行时 storage name、passing mode、ownership qualifier、tag/payload byte layout 和字段 byte offset/size/align；typed union 局部构造器 materialization、构造器到已有 typed union local 的简单赋值 materialization、struct 字段内 inline union 的构造器赋值与 `switch` 读取、`holder.inner.choice` 这类嵌套 struct 字段内 inline union 的构造器赋值与 `switch`/`using` 读取、inline `switch`/`using` tag/payload 读取、`switch`/`using` payload 绑定的声明类型传播与 owner payload 默认 borrow lowering、typed union 本地变量之间的声明初始化复制/简单赋值复制，以及 active variant owner-aware copy/drop/GC 分派已接入基础 runtime/compiler 路径。更完整复杂表达式/成员级 mutation 矩阵仍待后续阶段。
- **本阶段补充**：function type layout 现在会为 union variants 的 payload field metadata 预留容量；value-frame 参数目的槽复用时会先释放旧 owner，constructor carrier 转入 inline union 后会消费 carrier 内 owner payload 引用，避免 owner payload 经调用窗口或共享参数释放路径残留。子函数 typed union inline layout、active variant drop/replacement 与多 owner payload 的现有回归在 `zr_vm_union_test` 57/0 中通过。本次 explicit `move` slice 又补上 inline union payload pseudo-member 写回：`SET_MEMBER_SLOT_NULL` 可把已移动 payload 清空，后续 active variant drop 只看到 null owner slot。

## 7. 与 `enum` 的关系

- `enum` 保留不变（常量值，继承 `int/string/...`），用于纯标志/常量集。
- `union` 是带数据的代数类型。二者不合并：`enum` 是"命名常量"，`union` 是"带状态的 variant"。
- 退化情形：全 unit variant 的 `union`（如 `union Dir { N; S; E; W; }`）语义上接近 `enum`，但仍走 union 的 tag 布局与穷尽匹配——给用户一致的模式匹配体验。

## 8. 开放问题

- 模式匹配是否支持嵌套（`Some(Ok(x))`）与守卫条件（`Circle(r) if r > 0`）？建议首版支持一层解构 +  `if` 守卫如无必要则不支持，嵌套必须显式声明 `Optional.Some(x)`。
- owner variant 解构的默认语义是借用还是移动？建议**默认借用**（与 [01 篇](01-ownership-as-generics.md) `borrow` 一致），`move` 关键字显式移动。
- 穷尽性检查对泛型 union（`Option<T>`）在开放泛型上下文如何处理？按 variant 集合（与实参无关）检查，实参只影响绑定类型。
