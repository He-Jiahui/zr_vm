## 3. 分析与性能优化策略

### 3.1 错误恢复：补齐 panic-mode 同步点
现状 parser **没有显式的 panic-mode / 同步集**（`grep` 无 `recover`/`synchronize`）——一处语法错误易引发级联误报。本轮新增三类带块的语法（`union {...}`、扩展 `switch` 模式、`using ... else`），是引入轻量错误恢复的好时机：
- 在 `union` variant 列表、`switch` case 列表、`using` binder 解析失败后，**同步到稳定 token**（`;` / `}` / 下一个 variant 起始），各发一条诊断后继续，而非整体放弃。
- 同步集集中在 `parser_diagnostics.c`，提供 `parser_recover_to(set)` 辅助，避免每个 parser 函数各写一套。
- **收益**：一个文件里多个 union/match 错误能一次性报全，而不是只报第一个。

### 3.2 穷尽性检查：复用可达性，单趟收集
union `switch` 穷尽性**不另写数据流**：把 variant 全集作为"期望覆盖集"，遍历 case 时从集合划除，遍历后剩余集即"未覆盖 variant"，直接喂给 `match_non_exhaustive`。复用 `semantic_analyzer_reachability.c` 已有的分支可达框架，O(variant 数 + case 数) 单趟完成，无额外遍历。

### 3.3 借用检查：作用域树上的单趟标注
目标形态：`Borrow<T>`/`Loan<T>`/插件类型逃逸共用**同一套作用域逃逸分析**：
- 每个借用/owner 绑定记录其 lifetime region（已有 `TZrLifetimeRegionId`）。
- 赋值/返回/捕获到外层 region 时即 `*_escape` 诊断。
- 三类逃逸（borrow/loan/plugin）是同一分析的不同 `kind`，**一趟扫描出全部**，避免每类各扫一次。

当前落地范围（2026-06-18 02:19:58 +08:00）：
- 所有权泛型成员方法 `share()` / `weak()` / `borrow()` / `loan()` / `upgrade()` / `release()` / `detach()` 已由 type inference 按 receiver qualifier 校验并给出结果 qualifier，compiler lowering 直接发出 dedicated `OWN_*` opcodes；`using (owner.loan())` 已能在 cleanup 中解析本地 source owner 并归还。
- 赋值/变量初始化先通过 `ZrParser_AssignmentCompatibility_Check` 做 conservative 检查，Borrow 只允许流向 Borrow 目标，Loan 只允许流向 Loan 或 Borrow 目标；Unique/Shared 写入 plain 目标会报告 owner-to-plain 专用诊断。
- 源码级赋值表达式先通过 `ZrParser_AssignmentType_Infer` 复用同一 ownership flow 诊断；字段赋值 `box.slot = value` 已覆盖 plain 字段持久化边界。
- 调用参数兼容与 overload 解析失败路径会先尝试 ownership flow 诊断；plain 参数接收 Borrow/Loan/Unique/Shared 时会输出对应 escape，而不是普通 type mismatch/no overload。
- 普通泛型外壳的 ownership flow 诊断和 type compatibility 会递归检查 `elementTypes`；`Box<Borrow<T>>` / `Box<Loan<T>>` 会输出 borrow/loan escape，`Box<Unique<T>>` / `Box<Shared<T>>` 会输出 owner-to-plain escape，而不是因外层 canonical name 相同被放过。
- return 路径已有编译器侧基础 Borrow/Loan return 逃逸拦截，并与导出全局路径共用递归 `elementTypes` borrow/loan 扫描；尚未统一到 region fact/structured diagnostic 管线。
- 插件 guard 路径已有 compiler 侧基础 `plugin_type_escape` 拦截：`using (var p = %import(...))` block 内未 `share()` 的 `p`、callable member reference 或 block 内别名不能经 `return`、`throw`、`out`、普通调用参数、控制流条件中的调用参数、switch case 表达式中的调用参数、对象字段、数组元素、decorator metadata、parameter default metadata、signature type metadata 或 generator 延迟输出离开 guard，也不能被闭包或嵌套函数捕获；typed/untyped `DynamicModule<T>` 和内置 `PluginLoad.Available` `%import` guard 的 payload binding 也会在 block 前复用同一 scanner，return、throw、out、closure、call-argument、constructor-argument、switch-case call-argument、object-field、array-element、decorator/signature metadata、generator 负例矩阵均已覆盖；内层 block / `if` / loop 赋给外层局部的插件 alias 会在 region 退出时保留并继续参与后续 return/throw/out 扫描；`try` / `catch` / `finally`、generator/out、switch case value、declaration/member decorators、函数/方法/meta function parameter default/signature type 和带表达式 break/continue 中的同类逃逸路径也会继续扫描；guard-scoped module handle 的零参数 `.share()` 已作为显式 escape bridge 生成 shared owner；plain/default `@Available`/`PluginLoad.Available` guard 成功路径已为 guard scope 生成隐藏 shared owner cleanup，scope 退出发出 `OWN_RELEASE`；尚未统一到 semantic fact/LSP 结构化诊断管线。

### 3.4 签名比对：哈希先行，blob 兜底
[03 篇](03-metadata-and-token-model.md) 跨模块解析时，**先比 64-bit 签名哈希**（O(1) 拒绝绝大多数不匹配），仅哈希相等时才逐字节比对签名 blob 确认（防哈希碰撞）。插件守卫的运行期校验同理：哈希不匹配立即走 `else`，不做昂贵 blob 比较。

### 3.5 增量友好：检查产物进 fact 层
union/插件/所有权的语义结论（variant 覆盖、借用区间、签名解析结果）写入 **semantic fact 层**，使 LSP 增量重分析时可按需失效，而非整文件重算——与现有 semantic-fact-layer 的增量策略一致（见 [semantic-fact-layer.md](../../parser-and-semantics/semantic-fact-layer.md)）。

### 3.6 诊断去重与排序
新增检查可能在 parser 与语义层对同一位置各报一次（如 union variant 既语法错又语义不可达）。沿用结构化诊断的 location 去重：同 `location + code` 只保留 severity 最高的一条，按 location 排序输出，保证 LSP 与 CLI 体验一致。
