# LSP 与整体语义能力强化计划

本计划覆盖 ZR_VM 从「编译期语义推断」到「LSP / 调试器 / REPL」的端到端语义能力升级。核心思路一句话：**建一份以控制流图 + 数据流为引擎的统一语义事实层，让编译器、LSP、debug、REPL 共享同一推断与诊断**。

## 目标（对应用户诉求）

1. **编译期语义推断**：表达式类型、引用、数值、不可达代码、逻辑语义、所有权——统一到 CFG + dataflow 框架。→ [01](01-semantic-inference-core.md)
2. **token 追踪与引用关系**：强化 reaching-defs 引用推断，供导航/重命名复用。→ [01 §3](01-semantic-inference-core.md)
3. **错误表达能力**：不只说「预期 token」，要给「哪里错 / 为什么 / 怎么改」+ 关联位置 + 一键修复。→ [02](02-diagnostics-and-errors.md)
4. **LSP 行列定位修复**（UTF-16）+ 局部/增量推断 + 健壮性（不该推断时不崩）+ 基础能力完善。→ [03](03-lsp-robustness-and-position.md)
5. **调试器**：数据推断 / 表达式 / 条件断点能力强化（统一到共享语义层）。→ [04](04-debug-and-repl.md)
6. **REPL 表达式运行**：从「每次重编译、状态不持久」升级为常驻执行上下文。→ [04 §2](04-debug-and-repl.md)

## 重要前提（盘点结论）

调研后修正了几个初始假设——**不是从零做**：
- 诊断**已有** `code/message/cause/suggestion` 与 40+ 专用 builder，缺的是 relatedInformation、机器可应用 fix-it、类型诊断具体化、诊断码注册表。
- LSP 方法覆盖面**已很全**（hover/completion/definition/semanticTokens/codeAction/rename… 全注册），缺的是实现质量、行列正确性、局部推断、健壮性。
- 调试器**已支持**条件断点、hit-count、logpoint、表达式求值、变量检查，缺的是与主语义层统一、watchpoint、DAP。
- 语义事实模型（六类 fact）**结构齐全**，缺的是生产端（CFG/dataflow）与统一查询面。

详见 [00 · 现状盘点](00-current-state.md)。

## 里程碑（详见 05）

- **阶段 0**：UTF-16 行列修复 + 越界防护（无依赖，最先交付）。
- **阶段 1**：CFG + dataflow 引擎 + `semantic_query.h`（不可达诊断）。
- **阶段 2**：definite-assign / 所有权流 / 数值区间 + 诊断 related/fix。
- **阶段 3**：局部增量推断 + hover/completion 质量 + 诊断码注册表。
- **阶段 4**：debug 统一求值 / REPL 常驻上下文 / DAP。
