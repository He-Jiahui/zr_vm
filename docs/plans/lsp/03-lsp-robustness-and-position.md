## 1. 行列定位修复（UTF-16）—— 用户点名的 bug

### 1.1 病因（已确认）

`lsp_interface_position.c`：
- `ZrLanguageServer_Lsp_CalculateOffsetFromLineColumn`（`:82-117`）**按字节逐字节 `currentColumn++`**，把字节当列。
- `file_column_to_lsp_character` / `lsp_character_to_file_column`（`:63-69`）只做 +-1，把 LSP `character` 当文件列。
- `ZrLanguageServer_LspRange_ToFileRange`（`:195`）无 content 版直接 `offset = 0`。

LSP 规范：position 的 `character` 默认是 **UTF-16 code unit** 偏移（除非 `initialize` 时协商 `positionEncoding = utf-8 / utf-32`）。所以 λ(U+03BB，UTF-8 2 字节 / UTF-16 1 单元)、中文(UTF-8 3 字节 / UTF-16 1 单元)、😀(U+1F600，UTF-8 4 字节 / UTF-16 2 单元 surrogate) 全部错位。测试 `test_lsp_position_mapping.c:40-52` 已固化这两个 case。

### 1.2 修复方案

新建编解码层 `lsp_position_codec.c` / `.h`，所有「LSP position <-> 文件字节 offset」转换走这里，**列以 UTF-16 code unit 为单位**：

```
// 文件内容（UTF-8 字节）+ LSP position(line, utf16Char) -> 字节 offset
TZrSize ZrLsp_Utf16PositionToByteOffset(content, len, line, utf16Char);
// 字节 offset -> LSP position(line, utf16Char)
SZrLspPosition ZrLsp_ByteOffsetToUtf16Position(content, len, offset);
```

算法：
1. 先按 `\n` 定位行起始字节（保留现有行扫描）。
2. 在行内沿 UTF-8 解码每个码点：
   - 码点 < U+10000 -> UTF-16 占 1 单元；
   - 码点 >= U+10000 -> 占 2 单元（surrogate pair）。
3. 累加 UTF-16 单元数直到等于目标 `character`，返回当前字节 offset。反向同理。

并支持 `positionEncoding` 协商：`stdio_initialize.c` 在 capabilities 里声明 `positionEncoding`，若客户端选 `utf-8` 则编解码退化为字节计数（更快）。默认 utf-16。

### 1.3 改造点

- 重写 `CalculateOffsetFromLineColumn` 调用 `ZrLsp_Utf16PositionToByteOffset`。
- `LspRange_ToFileRange` / `LspPosition_ToFilePosition` 的「无 content」版标记为 deprecated，强制走带 content 版（offset 不能再填 0）。
- 反向（`SZrFileRange` -> `SZrLspRange`）也要按 UTF-16 重算 character，而非 `column-1`。已核对 parser 的 `SZrFilePosition.column` 语义是 1-based 字节列；带 content 的 LSP 输入路径先归一化到 byte offset，再生成 parser 风格的内部行/列。

### 1.4 验收

- `test_lsp_position_mapping.c` 现有 λ / emoji 用例通过。

## 2. 健壮性 —— 不该推断时不崩

原则：**LSP 任何请求处理都不得让进程崩溃**；推断失败 -> 返回空结果 / Unknown，并（可选）记一条诊断。

具体措施：
1. **越界防护**：`CalculateOffsetFromLineColumn`、`lsp_interface_support_file_position_from_offset` 等所有按 `content[index]` 访问处补 `index < contentLength` 检查（现状缺）。
2. **fallback AST 一致性**：`incremental_parser.c` 解析失败已保留旧 AST（`usesFallbackAst`），但要保证下游所有查询都尊重该标记——语义查询遇 fallback 一律降级为「仅词法/旧结果」。
3. **分析隔离**：01 篇的 CFG/dataflow 在 LSP 路径中包一层「分析守卫」：单函数分析 panic/越界（C 里用前置校验 + 失败返回码）不影响其它函数与其它请求。
4. **并发**：文件版本内容更新与读取需保证读到的是一致快照。当前已为中心转换 helper 提供 content snapshot acquire/free，避免转换期间直接持有 `fileVersion->content`；旧无 content public API 已删除，shared helper / `GetDocumentFilePosition` 无 snapshot 时不再调用旧 byte-column fallback；`lsp_interface.c`、`lsp_semantic_query.c`、`lsp_semantic_import_chain.c`、`lsp_token_metadata.c`、`lsp_semantic_tokens.c`、`lsp_folding_ranges.c`、`lsp_document_links.c`、`lsp_signature_help.c`、`lsp_code_action_imports.c`、`lsp_code_actions.c`、`lsp_hierarchy.c`、`lsp_editor_features.c`、`lsp_project_navigation.c`、`lsp_project.c`、`lsp_metadata_provider.c`、`incremental_parser.c` Parse、`stdio_document_color.c`、`stdio_completion.c`、`stdio_moniker.c`、`stdio_inline_completion.c`、`stdio_linked_editing.c`、`stdio_diagnostics.c`、`stdio_documents.c`、`stdio_inline_value.c`、`stdio_position_encoding.c` 和 `lsp_super_navigation.c` 的部分非转换读取已迁移到 snapshot；`incremental_parser.c` 内容存储/更新/释放和 snapshot ownership 边界已切到带 `contentGeneration` / `refCount` 的 `SZrFileVersionContentBlock`。生产源码复扫已无 `fileVersion->content` / `fileVersion->contentLength` direct residual；后续新增模块 direct content 读取复扫、WSL 补跑，以及更完整的跨请求并发审计仍未完成。
   2026-06-20 补充：当前 stdio 入口仍是顺序读消息、同步处理请求/通知的单线程模型；结合版本化/引用计数内容块和 direct content 残留清零，Stage 0 的当前跨请求并发审计已完成。此结论不覆盖未来新引入的多线程或异步请求入口。
5. **超时/规模上限**：超大文件或病态输入设节点数/时间上限，超限返回部分结果而非挂死。
6. **模糊测试**：对 position 编解码与增量更新加 fuzz（随机 UTF-8 + 随机编辑），断言不崩 + 往返一致。

### 3.1 增量解析（补完 TODO）

### 3.2 局部语义推断

LSP 的 hover/completion/definition 大多只关心**光标所在函数/作用域**。基于 01 篇 `semantic_query.h` 的「分析范围」参数：
- hover/typeAt：只对光标所在函数跑 CFG+推断，而非整模块。
- completion：只算光标作用域可见符号集（scope tree）。
- 失效粒度：编辑某函数体只让该函数的语义缓存失效，跨函数引用走符号表层。

借鉴 rust-analyzer 的 `Semantics` + 查询缓存思路（`lua/rust/src/tools/rust-analyzer/crates/hir/src/semantics.rs`），但用轻量版（按声明缓存 + 版本号失效），不引入完整 salsa。
